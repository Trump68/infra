аутентификации
Разрабатываемая ИС «Фармадок» должна обеспечивать возможность

Ролевая модель должна быть разработана в ходе технического проектирования

4.1.1.4. Модуль RAG
Модуль RAG является главным элементом системы, отвечающим за
семантический поиск данных в локальном хранилище. Модуль RAG состоит из двух
основных компонентов:
Retriever (компонент извлечения информации по запросу): извлекает данные
из векторной базы данных (Redis, Pinecone или аналог) на основе семантической
близости с учетом прав доступа, определенных в метаданных документов (уровень
конфиденциальности, RBAC-политики). При попытке доступа к данным без
соответствующих прав Retriever возвращает ошибку, которая логируется для аудита.
Generator (компонент генерации ответов): представляет собой большую
языковую модель, которая получает данные от Retriever (внутренние документы) и
агентов (внешние данные) для генерации ответа. В целях обеспечения
конфиденциальности обрабатываемых данных в качестве БЯМ генератора может
использоваться локальная модель, запущенная на сервере организации, или БЯМ,
развернутая в доверенном облачном сервисе.


4.1.1.5. Векторная база данных
Векторная база данных (Redis или аналог) хранит векторные вложения
(эмбеддинги) документов, созданные с использованием специализированных
моделей вложений (OpenAI Embeddings или аналог). Векторная база данных хранит
векторные эмбеддинги документов, зашифрованные с использованием AES-256 или
алгоритмов, соответствующих ГОСТ Р 34.12-2015 (в зависимости от требований
Заказчика).


Передача данных осуществляется по протоколу TLS версии 1.3. Ключи
шифрования хранятся в защищенном хранилище (например, HashiCorp Vault) с
автоматической ротацией каждые 90 дней.
Чувствительные данные перед индексацией в векторную базу данных
удаляются или маскируются (с помощью пакета Presidio или аналогов).

4.1.2. Требования к масштабируемости и производительности
ИС «Фармадок» должна обеспечивать работу не менее 100 одновременно
подключенных пользователей. Система должна поддерживать обработку не менее
100 запросов в минуту без потери функциональности.

Все данные, хранящиеся на
сервере, должны быть зашифрованы с использованием алгоритмов,
соответствующих стандартам ГОСТ Р 34.11-2012 и ГОСТ Р 34.12-2015.


Критическая информация, содержащаяся в загружаемых документах, должна
подвергаться автоматической анонимизации перед передачей в модель БЯМ. Для
этого в системе должен быть реализован промежуточный программный агент
(middleware), выполняющий проверку и очистку данных от персональных
идентификаторов в соответствии с требованиями конфиденциальности.



4.2.1.2. Семантический поиск с использованием модуля RAG
Система должна поддерживать семантический поиск информации в локальном
хранилище (векторной базе данных, например, Redis или Pinecone) с применением
технологии дополненного запроса (Retrieval-Augmented Generation, RAG).
Компонент Retriever извлекает релевантные данные на основе семантической
близости; компонент Generator (большая языковая модель, БЯМ) формирует ответ,
комбинируя извлеченные данные. Функция поддерживает обработку внутренних
документов и кэшированных результатов внешнего поиска.

4.2.1.3.
Технология получения сведений определяется на стадии технического проектирования.


4.2.1.4. Интеграция с векторной базой данных
Система обеспечивает хранение векторных эмбеддингов документов в
векторной базе данных (Redis или аналог) и обеспечение семантического поиска с
учетом прав доступа (RBAC). Эмбеддинги создаются с использованием моделей,
таких как OpenAI Embeddings, и шифруются по AES-256 или ГОСТ Р 34.12-2015.
Модуль RAG использует базу для извлечения данных (Retriever) и генерации
ответов (Generator).
