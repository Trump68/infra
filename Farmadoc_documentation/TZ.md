
# Изменение 1 -------------------------------------------------------------------

4.1.1.2. API Gateway
Основной точкой входа для всех пользователей является API Gateway (Kong
или аналог). API Gateway обрабатывает запросы, выполняет аутентификацию,
проверяет права доступа на основе ролевой модели (RBAC), реализует ограничение
частоты запросов (например, не более 100 запросов в минуту на пользователя) и
защищает от DDoS-атак с помощью WAF. Все запросы логируются для аудита с
сохранением метаданных (IP, пользователь, время, тип запроса).

%%
Обоснование изменения:

Формулировка о единственной точке входа через API Gateway не должна исключать типовую для веб-клиентов топологию с BFF (бэкенд для фронтенда): данный слой не отменяет функции шлюза, но может располагаться между клиентом и API Gateway. Требования к защите от DDoS относятся к периметру приёма трафика и сохраняют силу при любой согласованной схеме размещения BFF и API Gateway.

Предлагаемое изменение:

# Изменение 3 -------------------------------------------------------------------

4.1.1.4. Модуль RAG
Модуль RAG является главным элементом системы, отвечающим за
семантический поиск данных в локальном хранилище. Модуль RAG состоит из двух
основных компонентов:
Retriever (компонент извлечения информации по запросу): извлекает данные
из векторной базы данных (Redis, Pinecone или аналог) на основе семантической
близости с учетом прав доступа, определенных в метаданных документов (уровень
конфиденциальности, RBAC-политики). При попытке доступа к данным без
соответствующих прав Retriever возвращает ошибку, которая логируется для аудита.
Generator (компонент генерации ответов): представляет собой большую
языковую модель, которая получает данные от Retriever (внутренние документы) и
агентов (внешние данные) для генерации ответа. В целях обеспечения
конфиденциальности обрабатываемых данных в качестве БЯМ генератора может
использоваться локальная модель, запущенная на сервере организации, или БЯМ,
развернутая в доверенном облачном сервисе.

%%
Обоснование изменения:

Аббревиатура RAG (Retrieval-Augmented Generation) относится к паттерну «поиск + генерация», а не к одному программному модулю в смысле архитектуры поставки.
В техническом задании целесообразно описывать функциональность, видимую заказчику и подлежащую приёмке.
Разделение на компоненты Retriever и Generator в ТЗ создаёт лишнюю привязку к внутренней декомпозиции, которая может меняться (один сервис, два микросервиса, оркестратор и т.д.) без изменения требований. 
Указание конкретных продуктов (Redis, Pinecone) в ТЗ ограничивает реализацию и устаревает при смене стека.

Предлагаемое изменение:

# Изменение 4 -------------------------------------------------------------------

4.1.1.5. Векторная база данных
Векторная база данных (Redis или аналог) хранит векторные вложения
(эмбеддинги) документов, созданные с использованием специализированных
моделей вложений (OpenAI Embeddings или аналог). Векторная база данных хранит
векторные эмбеддинги документов, зашифрованные с использованием AES-256 или
алгоритмов, соответствующих ГОСТ Р 34.12-2015 (в зависимости от требований
Заказчика).

%%
Обоснование изменения:

Шифрование эмбеддингов не является общепринятой практикой, поскольку без расшифровки в рантайме индекс и поиск ломаются,
а если производить расшифровку в рантайме, то такое шифрование на уровне поля мало что дает. 
C точки зрения защиты от того, кто уже имеет доступ к этому процессу, памяти, отладчику, дампу — отдельное «шифрование поля» почти не добавляет барьера: ключ и расшифрованные данные уже там же, где и поиск.
Обычно используют шифрование диска (носителя), TLS + контроль доступа.

Предлагаемое изменение:

4.1.1.5. Векторная база данных
Векторная база данных (Redis или аналог) хранит векторные вложения
(эмбеддинги) документов, созданные с использованием специализированных
моделей вложений (OpenAI Embeddings или аналог). Носитель базы данных шифруется с использованием AES-256 или
алгоритмов, соответствующих ГОСТ Р 34.12-2015 (в зависимости от требований
Заказчика).

В ТЗ достаточно требований к доступу, метаданным и жизненному циклу данных, а обезличивание при необходимости задаётся отдельной политикой по классам сведений, а не абзацем про индексацию. Остальное можно регулировать с помощью политик RBAC.

# Изменение 8 -------------------------------------------------------------------

Критическая информация, содержащаяся в загружаемых документах, должна
подвергаться автоматической анонимизации перед передачей в модель БЯМ. Для
этого в системе должен быть реализован промежуточный программный агент
(middleware), выполняющий проверку и очистку данных от персональных
идентификаторов в соответствии с требованиями конфиденциальности.

%%
Обоснование изменения:

Необходимо уточнить, что критическая информация - это именно персональная информация или коммерческие секреты не предназначенные для обработки в Системе.
Кроме того, необходимо уточнить точку очистки от такого рода информации. 
Предлагается делать это уже при поступлении документа в систему, чтобы исключить любое хранение и использование персональной информации и коммерческих секретов.
Что касается другого рода чувствительной информации, которой нужно оперировать на определенном этапе, то это можно организовать с помощью RBAC,
и уточнить на этапе реализации RBAC.

Предлагается изменить:

Персональная информация, или коммерческие секреты, не предназначенные для обработки в Системе, подлежит обязательной очистке (анонимизации/удалению) на этапе поступления документа в систему — до ее хранения, обработки и передачи в модель БЯМ. Обработка иной чувствительной информации, необходимой на отдельных этапах бизнес-процессов, должна осуществляться с применением ролевой модели разграничения доступа (RBAC) с последующей детализацией соответствующих требований на этапе реализации.

# Изменение 17 -------------------------------------------------------------------

4.2.2.4. Добавление дополнительных агентов ИИ через штатный интерфейс
Система должна предоставлять штатный интерфейс для администраторов с
соответствующими правами доступа, позволяющий добавлять новые агенты ИИ в
качестве подключаемых модулей.
Новые агенты реализуются как плагины на основе фреймворков LangChain,
AutoGen или аналогов, и интегрируются без необходимости модификации основной
системы.
Каждый агент должен запускаться в изолированном контейнере (Docker) с
ограниченным доступом к сети (белый список доменов), файловой системе и
ресурсам.
Интерфейс должен поддерживать загрузку кода агента и настройку
параметров (например, доступ к API, лимиты ресурсов) и проверку безопасности
(сканирование на вредоносный код с использованием инструментов, таких как
ClamAV).

%%
Обоснование изменения:  
Неоднозначность трактования термина "интерфейс". Возможное дублирование функциональности управления докер-контейнерами и антивируса.

Предлагаемый вариант:

Новые агенты реализуются как плагины на основе фреймворков LangChain,
