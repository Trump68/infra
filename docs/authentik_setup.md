# Пошаговая настройка OAuth2/OpenID Provider в Authentik

Подробная инструкция по заполнению всех экранов мастера создания провайдера (Directory → Providers → Create → OpenID Connect Provider). Провайдер используется для входа браузерного приложения и проверки JWT в Kong.

---

## Шаг 1. Выбрать тип

- Выберите **OpenID Connect Provider** (OAuth2/OpenID Provider).
- Перейдите к следующему шагу.

---

## Шаг 2. Создать OAuth2/OpenID Provider

Мастер состоит из нескольких экранов. Ниже — каждый экран и рекомендуемые значения.

### 2.1. Основные параметры

| Поле | Что указать |
|------|------------------|
| **Имя*** | Произвольное имя провайдера (например `Farmadoc` или `default`). От него может зависеть **slug** в URL (например `default`), который понадобится для Kong. |
| **Поток аутентификации** | **default-authentication-flow (Welcome to authentik!)** — поток входа, когда пользователь ещё не аутентифицирован. |
| **Поток авторизации*** | **default-provider-authorization-explicit-consent (Authorize Application)** — Authorization Code flow с явным согласием; безопасный вариант. Не выбирайте implicit-consent (устаревший и небезопасный). |

Нажмите «Далее» или разверните блок **Настройки протокола**.

---

### 2.2. Настройки протокола — тип клиента

| Поле | Что выбрать |
|------|------------------|
| **Тип клиента*** | **Публичный (Public)** — если токены получает браузерное приложение (SPA). Секрет в браузере хранить нельзя, используется PKCE. Выберите **Конфиденциальный (Confidential)** только если с Authentik общается только ваш backend и хранит client secret. |

**ID клиента** — заполняется автоматически, менять не обязательно.

---

### 2.3. Перенаправляющие URI и подписывающий ключ

| Поле | Что указать |
|------|------------------|
| **Перенаправляющие URI/Источники (RegEx)*** | Точный URL, на который Authentik вернёт пользователя после входа. Например: `http://localhost:3000/callback` или `http://localhost:3000/auth/callback` (подставьте порт и путь вашего SPA). Несколько URI — с новой строки. Для продакшена **не** используйте `.*` (небезопасно). |
| **Подписывающий ключ** | Оставьте **authentik Self-signed Certificate**. |

---

### 2.4. Сроки действия кодов и токенов

| Поле | Рекомендация |
|------|------------------|
| **Срок действия кода доступа** | Оставить `minutes=1`. Код одноразовый, обменивается сразу. |
| **Срок действия Access токена** | По умолчанию `minutes=5` — часто требует обновления. Для разработки удобнее `hours=1` или `minutes=30`. Формат: `hours=1;minutes=0;seconds=0`. |
| **Срок действия Refresh токена** | Оставить `days=30` или задать, например, `days=90`. |

Формат везде: `hours=1;minutes=2;seconds=3` или `days=30`.

---

### 2.5. Области (Scopes)

- В **Выбранные области** должны быть минимум **openid** (обязателен для OIDC).
- Рекомендуется также **email** и **profile**, если приложению нужны почта и имя пользователя.
- Обычно по умолчанию уже выбраны нужные (например, три элемента OpenID) — ничего не меняйте и переходите дальше.
- Дополнительные scope добавляйте только если они реально нужны приложению.

---

### 2.6. Режим субъекта (Subject mode)

Определяет, что будет в утверждении `sub` в JWT.

| Вариант | Когда выбирать |
|---------|------------------|
| **На основе хэшированного идентификатора пользователя** | Рекомендуется по умолчанию: стабильный уникальный идентификатор, не раскрывает email/логин. |
| **На основе электронной почты пользователя** | Только если backend или приложение однозначно идентифицируют пользователя по email. |

Оставьте **хэшированный идентификатор**.

---

### 2.7. Утверждения в id_token и режим эмитента

| Поле | Что выбрать |
|------|------------------|
| **Включить утверждения в id_token** | **Включено** — утверждения из scope (openid, email, profile) попадут в id_token; не нужен отдельный запрос к userinfo. |
| **Режим эмитента*** | **У каждого провайдера есть свой эмитент, основанный на идентификаторе приложения.** Тогда issuer в токене будет вида `http://authentik-server:9000/application/o/<slug>/...`, что совпадает с настройкой в `kong/kong.yml`. Вариант «один идентификатор для всех провайдеров» для связки с Kong не используйте. |

---

### 2.8. Доверенные источники OIDC

- Если пользователи входят только через локальных пользователей Authentik (без LDAP и внешнего IdP) — **ничего не выбирайте**, оставьте 0 элементов.
- Добавляйте источники только если настроили в Authentik внешний источник (LDAP, другой OIDC) и хотите, чтобы через этот провайдер могли входить и пользователи оттуда.

---

### 2.9. Завершение

Нажмите **Завершить**. Запомните **slug** созданного провайдера (виден в списке провайдеров и в URL при редактировании). Он понадобится для Kong и для создания Application.

---

## После создания провайдера

### Создание приложения (Application)

**Applications** → **Create** — откроется форма «Создание приложения». К приложению привязывается созданный OAuth2/OIDC провайдер; Redirect URIs задаются в провайдере, здесь указывается только связь приложение–провайдер.

| Поле | Что указать |
|------|------------------|
| **Имя*** | Отображаемое название приложения (например `farmadoc_client` или `Farmadoc SPA`). |
| **Идентификатор*** | Внутренний slug для URL (латиница, без пробелов). Часто совпадает с именем, например `farmadoc_client`. |
| **Группа** | По желанию — имя группы, чтобы приложения отображались сгруппированными (например `farmadoc`). Можно оставить пустым. |
| **Провайдер*** | Выберите созданный OAuth2/OIDC провайдер (тот, у которого настроены Redirect URIs и тип клиента). |
| **Поставщики обратных каналов** | Оставьте пустым (не добавляйте провайдеров), если не используете SAML, SCIM и т.п. |
| **Режим механизма политики*** | **any** («Любая политика должна соответствовать») — достаточно для обычного входа. **all** — только если доступ должен даваться только при выполнении всех выбранных политик. |
| **Настройки пользовательского интерфейса** | Опционально: иконка, ссылка на приложение. Для связки Kong + SPA можно не заполнять. |

Нажмите **Создать**. Redirect URIs и остальные параметры OAuth2 задаются в настройках **провайдера**, не приложения.

### URL для Kong (issuer)

Kong проверяет входящие JWT по OpenID Connect: сверяет подпись токена и утверждение `iss` (issuer) с тем, что возвращает Authentik. Параметр **issuer** в конфиге Kong — это URL документа **OpenID Connect Discovery** вашего провайдера. По этому URL Kong получает метаданные (в т.ч. JWKS для проверки подписи) и сверяет значение `iss` в токене.

**Формат URL:**
```
http://authentik-server:9000/application/o/<slug>/.well-known/openid-configuration/
```
- **`<slug>`** — идентификатор (slug) провайдера в Authentik. Не путать с именем приложения: slug у провайдера (Directory → Providers → ваш провайдер → в URL или в карточке будет что-то вроде `farmadoc_default` или `default`).
- **`authentik-server:9000`** — хост и порт Authentik **из сети Docker** (Kong и Authentik в одной сети `farmadoc-network`, поэтому используется имя сервиса). Если позже будете использовать внешний домен с TLS, в issuer нужно будет указать тот же URL, который Authentik пишет в токены (иначе проверка `iss` не пройдёт).

**Где править:** файл `kong/kong.yml` в корне репозитория. Блок:
```yaml
plugins:
  - name: openid-connect
    config:
      issuer: http://authentik-server:9000/application/o/default/.well-known/openid-configuration/
```
Замените в `issuer` сегмент после `/o/` на slug вашего провайдера. Например, если slug = `farmadoc_default`:
```yaml
issuer: http://authentik-server:9000/application/o/farmadoc_default/.well-known/openid-configuration/
```
Если slug = `default`, строка в текущем примере уже подходит.

**После изменений:** перезапустите Kong, чтобы подхватить конфиг: `docker compose up -d kong` или `docker compose restart kong`.

**Проверка issuer:** в браузере (или `curl`) откройте получившийся URL — должен вернуться JSON с полями `issuer`, `jwks_uri`, `authorization_endpoint` и др. Kong обращается к этому URL из контейнера, поэтому с хоста проверяйте через `http://localhost:9000/application/o/<slug>/.well-known/openid-configuration/` (порт 9000 проброшен на хост).

### Проверка

Запрос к `http://localhost:8001/api/...` без токена должен вернуть 401; с валидным Bearer-токеном от Authentik — ответ от backend.

## Автоматизация: скрипты, Blueprints и API

Настройку можно не делать вручную в UI, а выполнить программно.

### 1. Blueprints (YAML)

Authentik поддерживает **Blueprints** — YAML-описание провайдеров, приложений и потоков. Файл можно:

- положить в каталог `/blueprints` в контейнере (смонтировать volume) — Authentik подхватывает его периодически;
- загрузить через веб: **Customization** → **Blueprints** → **Apply blueprint**;
- применить через API (см. ниже готовый blueprint Farmadoc).

В blueprint задают модель `authentik_providers_oauth2.oauth2provider` (провайдер) и `authentik_core.application` (приложение), ссылаясь на существующие потоки и ключ подписи через `!Find`. Потоки вроде `default-authentication-flow` и `default-provider-authorization-explicit-consent` создаются при первом запуске Authentik.

#### Готовый blueprint Farmadoc (применение через API)

В репозитории есть файл **`authentik/blueprints/farmadoc-oidc.yaml`**: создаёт провайдер OAuth2/OIDC «Farmadoc OIDC» и приложение «farmadoc_client» с настройками для Kong и SPA (публичный клиент, explicit consent, redirect `http://localhost:3000/callback`, scopes openid/email/profile).

**Перед применением:** выполнен первичный вход в Authentik (initial-setup), существуют потоки по умолчанию и ключ «authentik Self-signed Certificate».

**Способ 1 — через веб-интерфейс:** **Customization** → **Blueprints** → **Apply blueprint** → загрузить файл `authentik/blueprints/farmadoc-oidc.yaml`.

**Способ 2 — через API:**

1. Создать API-токен: **System** → **Tokens** → создать токен с правами на управление blueprints (и при необходимости core/providers).
2. Создать экземпляр managed blueprint с содержимым YAML и применить его.

   Пример (подставьте `$HOST`, `$TOKEN` и при необходимости скорректируйте путь к файлу):

   ```bash
   # Базовый URL Authentik (с хоста, например после SSH-туннеля)
   AUTHENTIK_URL="http://localhost:9000"
   TOKEN="your-api-token"

   # Создать managed blueprint: тело запроса — JSON с полем path или content (зависит от версии API).
   # Вариант с загрузкой содержимого файла как body (если API принимает YAML в теле):
   curl -s -X POST "${AUTHENTIK_URL}/api/v3/blueprints/instances/" \
     -H "Authorization: Bearer ${TOKEN}" \
     -H "Content-Type: application/json" \
     -d "{\"path\": \"authentik/blueprints/farmadoc-oidc.yaml\"}"  # или передать content с телом YAML

   # Либо: если в вашей версии Authentik есть «flow import» по URL/файлу, используйте соответствующий эндпоинт.
   # После создания экземпляра применить его (подставьте UUID из ответа создания):
   # curl -s -X POST "${AUTHENTIK_URL}/api/v3/blueprints/instances/<UUID>/apply/" \
   #   -H "Authorization: Bearer ${TOKEN}"
   ```

   Точные имена полей и эндпоинты см. в Swagger вашего инстанса: `http://localhost:9000/api/v3/schema/swagger/`. В части версий используется `POST /managed/blueprints/` (тело — метаданные и путь/контент blueprint), затем `POST /managed/blueprints/:uuid/apply/`.

3. После успешного применения в **Directory** → **Providers** появится провайдер «Farmadoc OIDC», в **Applications** — приложение «farmadoc_client». Slug провайдера (в URL, например `farmadoc-oidc`) подставьте в `kong/kong.yml` в параметр `issuer` (см. раздел «URL для Kong (issuer)» выше).

**Изменение redirect URI:** при необходимости отредактируйте в `farmadoc-oidc.yaml` массив `redirect_uris` (например добавьте `https://your-app.example/callback`) и примените blueprint заново или измените провайдер в UI.

### 2. REST API

Провайдеры и приложения можно создавать через REST API Authentik:

- Токен API: в веб-интерфейсе **System** → **Tokens** создать токен с правами на создание провайдеров и приложений.
- Документация API: после входа в панель — `http://localhost:9000/api/v3/schema/swagger/` (или `/redoc/`).
- Основные эндпоинты: создание провайдера (OAuth2), создание приложения, привязка провайдера к приложению. Порядок вызовов важен: сначала провайдер, затем приложение с указанием `provider` по ID.

Скрипт может вызывать API через `curl` или любой HTTP-клиент (Python, etc.), передавая заголовок `Authorization: Bearer <token>`.

### 3. Итог

| Способ | Плюсы | Минусы |
|--------|--------|--------|
| **UI (ручная настройка)** | Понятно, все поля под рукой | Долго, не воспроизводимо |
| **Blueprint (YAML)** | Версионирование, повторяемость, «инфраструктура как код» | Нужно знать slug потоков и структуру YAML |
| **REST API** | Гибкая автоматизация, свои скрипты | Нужен токен и знание схемы API |

Для полностью скриптовой настройки после первого запуска Authentik разумно использовать **Blueprint**: один раз подготовить YAML под ваши потоки и slug, положить в volume или применить через API.
