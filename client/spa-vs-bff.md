# Статический SPA и SPA + BFF: сравнение и безопасность

Документ поясняет разницу между двумя способами развёртывания браузерного приложения с OIDC (Kong + Authentik) и отвечает на вопрос, какой вариант безопаснее.

---

## Что такое статический SPA

**Статический SPA** — это одностраничное приложение, которое:

- Раздаётся как набор статических файлов (HTML, JS, CSS) с CDN или веб-сервера (nginx, S3 и т.п.).
- **Не имеет своего backend-сервера** приложения: весь код выполняется в браузере.
- Для входа по OIDC: редирект на Authentik → callback с `code` → **обмен code на токен выполняется в браузере** (fetch на token endpoint Authentik с PKCE).
- Запросы к API: браузер **напрямую** обращается к Kong (или к другому API) с заголовком `Authorization: Bearer <token>`.

Чтобы это работало, нужны:

- **CORS** на token endpoint Authentik (разрешить origin вашего SPA).
- **CORS** на Kong (или API и ваш SPA на одном домене через reverse proxy).

Клиент в Authentik при этом обязательно **публичный** (public): без `client_secret`, с PKCE. Секреты в браузере хранить нельзя.

---

## Что такое SPA + BFF

**SPA + BFF** (Backend for Frontend) — это та же статика плюс **отдельный сервер приложения** (BFF), который:

- Раздаёт статические файлы SPA и отдаёт конфиг (например `GET /config.json`).
- Принимает от SPA запрос «обменять code на токен» (`POST /auth/exchange`) и **сам** ходит в Authentik (token endpoint), передаёт `code`, `code_verifier` (PKCE), при необходимости `client_secret` (если клиент confidential).
- Проксирует запросы к API: SPA вызывает **тот же origin** (например `GET /api/...`), BFF подставляет токен и пересылает запрос в Kong.

Браузер **не обращается напрямую** ни к Authentik (кроме редиректа на страницу входа), ни к Kong. Всё взаимодействие с IdP и API идёт через BFF.

---

## Сравнение

| Критерий | Статический SPA | SPA + BFF |
|----------|-----------------|-----------|
| **Раздача приложения** | Только статика (CDN, nginx) | Статика + сервер приложения (тот же или отдельный хост) |
| **Обмен code → token** | В браузере (fetch на Authentik) | На BFF; браузер шлёт только code и code_verifier |
| **Вызовы API** | Браузер → Kong (напрямую или через общий proxy) | Браузер → BFF → Kong |
| **CORS** | Нужен на Authentik и Kong для origin SPA | Не нужен к Authentik/Kong из браузера (браузер общается только с BFF) |
| **Секреты (client_secret)** | Невозможны: только public client + PKCE | Возможны: BFF может быть confidential client |
| **Где хранится токен** | В браузере (память, sessionStorage) | В браузере или в httpOnly cookie (BFF держит сессию) |
| **Видимость токена для JS** | Да (если в памяти/sessionStorage) | По желанию: можно не отдавать токен в JS (cookie, BFF подставляет в proxy) |
| **Зависимость от инфраструктуры** | Зависит от CORS и доступности IdP/API с браузера | Зависит только от доступности BFF; IdP/API могут быть во внутренней сети |

## Безопасность: по пунктам

### 1. Токен в браузере

- **Статический SPA:** токен неизбежно попадает в JS (память или sessionStorage). Любой XSS может прочитать его и отправить наружу.
- **SPA + BFF:** токен может оставаться только на BFF (сессия, cookie с httpOnly). Браузер тогда не получает токен в JS — только cookie; при запросах к `/api` BFF сам подставляет токен. XSS не сможет украсть токен из cookie (если не ломать httpOnly).  
  Вариант «BFF отдаёт токен в теле ответа, SPA кладёт в память» по риску XSS аналогичен статическому SPA.

**Вывод:** SPA + BFF даёт возможность **не светить токен в JS** (cookie + proxy) и тем самым снизить последствия XSS. Статический SPA такой возможности не имеет.

### 2. Секреты (client_secret)

- **Статический SPA:** клиент может быть только **публичным**. `client_secret` в браузере хранить нельзя — его узнают все пользователи.
- **SPA + BFF:** BFF может использовать **confidential** клиент и хранить `client_secret` в env/секрет-менеджере. Перехват code злоумышленником без `code_verifier` (PKCE) или без `client_secret` не даёт токен. Дополнительная защита по сравнению с «только PKCE» в публичном клиенте.

**Вывод:** SPA + BFF позволяет при желании использовать confidential client и не полагаться только на PKCE. Со статическим SPA только PKCE.

### 3. CORS и доступность IdP/API

- **Статический SPA:** браузер должен иметь прямой доступ к token endpoint Authentik и к Kong. Нужна настройка CORS; token endpoint и API оказываются в «публичной» зоне с точки зрения браузера.
- **SPA + BFF:** Authentik и Kong могут быть **внутренними** (не открыты в браузере). Браузер общается только с BFF; BFF уже из доверенной сети ходит в IdP и Kong. Меньше поверхность атаки с интернета.

**Вывод:** SPA + BFF безопаснее с точки зрения размещения IdP и API (можно не светить их в CORS и не открывать наружу).

### 4. Поверхность атаки

- **Статический SPA:** атакующий может пытаться вызывать token endpoint и API напрямую (если знает URL и обходит CORS), подставлять свой code и т.д. Зависимость от правильной настройки CORS и политик IdP/API.
- **SPA + BFF:** прямых вызовов к IdP и Kong из браузера нет. Атаки идут в первую очередь на BFF и на сам SPA (XSS, CSRF). BFF можно усилить (rate limit, проверка origin, сессии).

**Вывод:** SPA + BFF концентрирует трафик на одном прикладном сервере (BFF), что упрощает контроль и мониторинг; IdP и API не светятся браузеру.

### 5. Целостность конфигурации

- **Статический SPA:** URL IdP, client_id, scope и т.д. либо зашиты в билд, либо подгружаются с какого-то публичного endpoint. При утечке конфига злоумышленник знает, куда стучаться и с каким client_id (для public клиента это допустимо, но даёт информацию).
- **SPA + BFF:** конфиг можно отдавать только с BFF (`/config.json`) и при необходимости привязывать к сессии/пользователю. Секреты (если есть) на BFF не отдаются.

**Вывод:** SPA + BFF даёт больший контроль над тем, кто и что узнаёт о конфигурации OIDC.

---

## Итоговые выводы

1. **Безопаснее в типичном случае — SPA + BFF:**
   - Токен можно не отдавать в JS (httpOnly cookie + proxy), что снижает риск при XSS.
   - Можно использовать confidential client и хранить `client_secret` только на BFF.
   - IdP и API не обязаны быть доступны и не обязаны отдавать CORS из браузера; их можно спрятать за BFF.
   - Вся логика обмена токена и вызова API сосредоточена на BFF — проще аудит и защита (rate limit, проверки, логи).

2. **Статический SPA** проще по развёртыванию (только статика и CORS), но:
   - Токен всегда доступен JS (риск XSS).
   - Только публичный клиент и PKCE; нет возможности использовать client_secret.
   - IdP и API должны быть доступны браузеру и корректно настроены по CORS.

3. **Когда уместен статический SPA:** максимально простой деплой, нет своего backend, при этом вы готовы мириться с хранением токена в JS и только с PKCE. IdP и API при этом должны поддерживать CORS и быть рассчитаны на публичный доступ с браузера.

4. **Когда предпочтителен SPA + BFF:** продакшен, повышенные требования к безопасности, желание не светить токен в JS и не открывать IdP/API в CORS. В этом репозитории реализован именно этот вариант в каталоге client/ (BFF в `serve.py`, конфиг из env, PKCE, проксирование `/api` в Kong).

## Зачем при SPA + BFF нужен Kong? Он излишен?

В схеме SPA + BFF поток к API такой: **браузер → BFF → Kong → backend**. Kong в этой цепочке не обязателен «по определению», но и не лишний — всё зависит от того, где вы хотите проверять JWT и держать функции шлюза.

### Что делает Kong в цепочке SPA + BFF

- **Проверяет JWT** (подпись, issuer, срок действия) и до backend пускает только запросы с валидным токеном.
- Работает как **единая точка входа** к API (один адрес для разных клиентов).
- Может давать **функции API Gateway**: лимиты запросов, маршрутизация, логирование и т.д.

То есть Kong — это «охранник» перед backend: без валидного токена от Authentik запрос до backend не доходит.

### Когда Kong можно считать избыточным

Если у вас **только этот SPA и один BFF** как единственный клиент backend:

- BFF **уже получил** токен от Authentik (через code + PKCE) и при желании может **сам** проверять JWT (например, по JWKS Authentik).
- После проверки BFF может вызывать backend **напрямую**, без Kong.

Тогда цепочка будет: **браузер → BFF → backend**. Роль «проверка JWT + проксирование» переносится в BFF, Kong для этого одного сценария не нужен — в этом смысле он **избыточен**.

### Когда Kong не излишен

Kong имеет смысл оставлять, если:

1. **Несколько разных клиентов** одного и того же API: не только этот SPA (через BFF), но и мобильное приложение, другие сервисы. Тогда один Kong — единое место, где проверяется JWT и решается, пускать ли запрос к backend. Без Kong каждому клиенту или каждому backend пришлось бы самому заниматься проверкой токенов.
2. **Backend не должен знать про OIDC/JWT**: вы хотите, чтобы backend только получал уже «очищенные» запросы (например, с заголовками вроде `X-User-Id`), а проверку подписи и issuer делал один компонент — им удобно сделать Kong.
3. **Нужны возможности API Gateway**: лимиты по ключу/пользователю, маршрутизация по путям/хостам, централизованные логи, политики. Kong для этого и предназначен; в BFF всё это дублировать неохота.

В таких случаях Kong не излишен, а полезен.

### Краткий вывод

- **Зачем при SPA + BFF нужен Kong:** чтобы в одном месте проверять JWT и пускать к backend только валидные запросы, плюс при необходимости давать функции шлюза (лимиты, маршрутизация и т.д.).
- **Он излишен?** Только в узком случае: один SPA, один BFF, один backend, и вы готовы перенести проверку JWT в BFF и звать backend напрямую. Тогда Kong для этой одной цепочки можно убрать.
- **Он не излишен**, если у вас несколько клиентов API, вы не хотите трогать backend под OIDC или хотите централизовать проверку токенов и политики в одном шлюзе — тогда Kong на своём месте.
