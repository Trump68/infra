## 1. Назначение документа

Настоящий документ описывает программное обеспечение ИС «Фармадок» на уровне логики работы кода, состава модулей, их взаимодействия, входных и выходных данных.

Документ предназначен для:

## 2. Назначение программного обеспечения

ПО ИС «Фармадок» предназначено для автоматизации обработки и анализа фармацевтической документации с использованием ИИ-компонентов и семантического поиска.

Ключевые функции ПО:
- загрузка, хранение и разметка документов;
- поиск релевантной информации по внутреннему корпусу документов (RAG);
- анализ документов на соответствие требованиям;
- сравнение документов и подготовка выводов;
- формирование отчетов в формате DOCX;
- обеспечение авторизованного доступа, аудита и наблюдаемости.

---

## 3. Логическая структура программного обеспечения

ПО построено по модульному принципу и включает следующие программные компоненты:

1. пользовательский веб-интерфейс;
2. API Gateway;
3. подсистема аутентификации и авторизации (IdP + BFF + проверка RBAC);
4. backend-сервисы прикладной логики;
5. модуль парсинга и разметки документов;
6. модуль RAG (Retriever/Generator);
7. модуль агентов ИИ;
8. модуль генерации отчетов DOCX;
9. подсистема хранения и обмена (PostgreSQL, S3 MinIO, Milvus, RabbitMQ);
10. подсистема аудита, логирования и мониторинга.

Принципы реализации:
- модульность и расширяемость за счет сервисной декомпозиции;
- изоляция агентов в контейнерной среде;
- единая точка входа через API Gateway;
- обработка запросов с учетом ролей доступа (RBAC);
- защита данных в передаче и хранении.

---

## 4. Описание модулей и их взаимодействия

### 4.1. Пользовательский веб-интерфейс

Назначение:
- предоставляет экранные формы для загрузки документов, поиска, анализа и формирования отчетов;
- инициирует пользовательские сценарии и отображает результаты.

Входные данные:
- действия пользователя;
- параметры поиска/анализа;
- файлы документов (DOCX, PDF, изображения).

Выходные данные:
- запросы в BFF/API;
- отображение результатов поиска, анализа и статусов обработки.

### 4.2. API Gateway

Назначение:
- единая точка входа для backend API;
- централизованная проверка токенов, маршрутизация, ограничение частоты запросов;
- журналирование запросов для аудита.

Входные данные:
- HTTP(S)-запросы от BFF/клиентских компонентов.

Выходные данные:
- маршрутизированные запросы к backend-сервисам;
- записи технических логов/метрик.

### 4.3. Подсистема аутентификации и авторизации

Состав:
- IdP (Authentik как базовый вариант реализации; Keycloak как допустимая альтернатива по согласованию);
- BFF-компонент;
- механизм ролевой проверки (RBAC) при доступе к API и данным.

Назначение:
- аутентификация пользователя (SSO/MFA);
- выпуск и проверка JWT;
- применение ролевой модели доступа.

Входные данные:
- учетные данные пользователя и факторы MFA;
- токены/коды авторизации OAuth2/OIDC;
- матрица ролей и правил доступа.

Выходные данные:
- утвержденная пользовательская сессия;
- JWT с утверждениями о пользователе/ролях;
- решения доступа (разрешено/запрещено).

### 4.4. Backend-сервисы прикладной логики

Назначение:
- реализуют бизнес-операции системы;
- координируют работу модулей хранения, RAG, агентов и генерации отчетов;
- предоставляют API для пользовательских сценариев.

Входные данные:
- валидированные запросы от API Gateway;
- данные из БД/хранилищ;
- события из очередей.

Выходные данные:
- структурированные ответы API;
- команды в подсистемы обработки и хранения;
- события и записи аудита.

### 4.5. Модуль парсинга и разметки документов

Назначение:
- извлечение текста/структуры из входных документов;
- подготовка фрагментов для индексации;
- формирование метаданных документов.

Входные данные:
- файлы из объектного хранилища;
- задания на обработку.

Выходные данные:
- нормализованные текстовые фрагменты;
- метаданные в PostgreSQL;
- данные для индексации в Milvus.

### 4.6. Модуль RAG (Retriever/Generator)

Назначение:
- поиск релевантных фрагментов по эмбеддингам;
- генерация ответа на основе найденного контекста;
- фильтрация выдачи по правам доступа.

Входные данные:
- поисковый запрос пользователя;
- коллекции эмбеддингов и индексов;
- метаданные документов и права доступа.

Выходные данные:
- релевантные фрагменты и ссылки на источники;
- сформированный ответ для пользовательского интерфейса/отчета.

### 4.7. Модуль агентов ИИ

Назначение:
- выполнение специализированных сценариев (проверка соответствия, сравнение документов, разметка, дополнительные задачи);
- запуск в изолированных контейнерах с ограничениями среды.

Входные данные:
- задания оркестратора/backend;
- документы и контекстные данные;
- параметры запуска агента.

Выходные данные:
- результаты выполнения задачи (структурированные выводы, артефакты);
- технические события выполнения и ошибки.

### 4.8. Модуль генерации отчетов DOCX

Назначение:
- формирование итоговых отчетов по результатам анализа и сравнения;
- сохранение отчетов в объектном хранилище и выдача пользователю.

Входные данные:
- шаблоны документов;
- структурированные результаты анализа.

Выходные данные:
- файлы отчетов DOCX;
- ссылки/идентификаторы сформированных отчетов.

### 4.9. Подсистема хранения и обмена

Состав и роль компонентов:
- `PostgreSQL` - операционные данные, метаданные документов, прикладной аудит;
- `S3 MinIO` - исходные документы и сформированные отчеты;
- `Milvus` - эмбеддинги и векторные индексы для поиска;
- `RabbitMQ` - асинхронный обмен заданиями/событиями между сервисами.

Входные данные:
- данные прикладных сервисов;
- файлы, события, результаты обработки.

Выходные данные:
- сохраненные сущности и объекты;
- сообщения для связанных модулей;
- выборки для API и RAG.

### 4.10. Подсистема аудита, логирования и мониторинга

Назначение:
- сбор и хранение технических и прикладных событий;
- поддержка трассировки и контроля инцидентов;
- передача критичных событий в SIEM (в составе поставки).

Входные данные:
- логи приложений и инфраструктуры;
- события безопасности и аудита.

Выходные данные:
- централизованные журналы;
- метрики и сигналы алертинга;
- выгрузки событий в SIEM.

## 5. Основные программные потоки

### 5.1. Сценарий входа и доступа к API

1. Пользователь инициирует вход через интерфейс.
2. BFF перенаправляет пользователя в IdP и получает результат аутентификации.
3. IdP выпускает JWT с ролями.
4. BFF/API Gateway выполняют проверку токена и правил RBAC.
5. Запрос передается в backend-сервис при положительном решении доступа.

### 5.2. Сценарий загрузки и индексации документа

1. Пользователь загружает файл через веб-интерфейс.
2. Backend сохраняет файл в `S3 MinIO`, метаданные - в `PostgreSQL`.
3. Через `RabbitMQ` формируется задание на парсинг/разметку.
4. Модуль обработки извлекает текст, формирует фрагменты и эмбеддинги.
5. Индексы/векторы сохраняются в `Milvus` с привязкой к метаданным и правам.

### 5.3. Сценарий семантического поиска и ответа

1. Пользователь отправляет запрос.
2. Retriever выполняет поиск в `Milvus` с учетом RBAC-фильтрации.
3. Generator формирует ответ на базе найденного контекста.
4. Результат и ссылки на источники возвращаются в интерфейс.

### 5.4. Сценарий формирования отчета

1. Пользователь запускает задачу анализа/сравнения.
2. Модуль агентов/аналитики формирует структурированный результат.
3. Модуль отчетности создает DOCX по шаблону.
4. Отчет сохраняется в `S3 MinIO` и становится доступным пользователю.

## 6. Входные и выходные данные по типам

Входные данные:
- документы: DOCX, PDF, изображения;
- пользовательские запросы и параметры сценариев;
- учетные данные/токены аутентификации;
- служебные события и сообщения очередей.

Выходные данные:
- результаты семантического поиска и аналитики;
- сформированные отчеты DOCX;
- метаданные, журналы аудита и технические логи;
- уведомления о статусах обработки.

---

## 7. Используемые программные средства и библиотеки (уровень проекта)

В документируемой конфигурации применяются:
- backend: Python 3.11+ (FastAPI или эквивалент);
- фронтенд: веб-клиент (SPA/MPA в зависимости от реализации);
- API Gateway: Kong OSS (или функциональный аналог);
- IdP: Authentik (базовый вариант) или Keycloak по согласованию;
- хранилища и обмен: PostgreSQL, S3 MinIO, Milvus, RabbitMQ;
- контейнеризация и изоляция сервисов: Docker (при необходимости оркестрация).

Конкретные версии компонентов фиксируются в эксплуатационно-техническом контуре поставки и конфигурационных файлах реализации.

## 8. Требования к сопровождению и развитию ПО

- Документ подлежит актуализации при изменении логики модулей, API-контрактов и потоков данных.
- Расширение функционала выполняется преимущественно через подключаемые модули/агенты без изменения ядра.
- Изменения в механизмах доступа и безопасности синхронизируются с матрицей RBAC, политиками аудита и журналирования.
- Новые функции второй очереди (безопасный веб-поиск, перевод, OCR, управление плагинами) документируются как расширение текущей структуры ПО.
