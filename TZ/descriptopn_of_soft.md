## 1. Введение

### 1.1. Назначение программы

Настоящий документ описывает программное обеспечение информационной системы «Фармадок», предназначенной для автоматизации поиска, обработки и анализа фармацевтической документации с использованием технологий искусственного интеллекта на основе больших языковых моделей (см. раздел 2 Технического задания на создание ИС «Фармадок»). Система ориентирована на использование в ФГБУ «НЦЭСМП» Минздрава России, а также в фармацевтических компаниях, регуляторных органах и организациях, занимающихся оценкой соответствия и контролем качества лекарственных средств.

Описание выполнено в соответствии с ГОСТ 19.402-78 и ЕСПД (ГОСТ 19 серии) и предназначено для разработчиков (доработка и сопровождение кода) и инженеров по валидации ПО.

### 1.2. Условные обозначения и сокращения

Условные обозначения и сокращения приведены в разделе 1 Технического задания на создание ИС «Фармадок». Основные: БЯМ (большая языковая модель), RAG (Retrieval-Augmented Generation), RBAC (Role-Based Access Control), API (Application Programming Interface), DOCX (формат файлов Microsoft Word), MFA (Multi-Factor Authentication), НЦЭСМП (Научный центр экспертизы средств медицинского применения), ПУР (план управления рисками), TLS (Transport Layer Security), WAF (Web Application Firewall).

## 2. Логическая структура программы

### 2.1. Общая структура

Программный комплекс ИС «Фармадок» представляет собой совокупность логических модулей (компонентов), взаимодействующих по заданным правилам. Описание даётся без привязки к физическим серверам — только логические модули и потоки данных между ними.

**Основные логические компоненты:**

- Клиентская часть (пользовательский веб-интерфейс) — одностраничное приложение, выполняемое в браузере.
- API Gateway — единая точка входа для всех запросов от клиента к серверной части.
- Сервисы backend: аутентификация и авторизация, модуль RAG (Retriever и Generator), модуль агентов ИИ, модуль парсинга и разметки документов, модуль генерации отчётных документов.
- Подсистема работы с векторной базой данных (хранение и поиск эмбеддингов документов).
- Взаимодействие с большой языковой моделью (БЯМ) для генерации ответов.

Состав модулей согласован с документом «Описание архитектуры» (Architecture.md), раздел 6. Детальное описание каждого модуля приведено в разделе 3 настоящего документа.

### 2.2. Взаимодействие модулей

Потоки управления и данных между модулями организуются следующим образом.

- **Клиент → API Gateway:** все запросы пользователя (поиск, загрузка документов, запуск анализа/сравнения, выгрузка отчётов) поступают в систему через API Gateway по протоколу HTTPS.
- **API Gateway → сервисы backend:** после проверки аутентификации (JWT) и прав доступа (RBAC) запрос маршрутизируется на соответствующий сервис (аутентификация, RAG, агенты, генерация отчётов и т.д.).
- **Модуль RAG ↔ векторная база данных:** Retriever выполняет запросы к векторной БД (поиск по эмбеддингам); результаты индексации документов записываются в векторную БД модулем парсинга/индексации.
- **Модуль RAG ↔ БЯМ:** Generator передаёт контекст и запрос в большую языковую модель и получает сгенерированный ответ.
- **Модуль агентов ↔ модуль RAG:** агенты (сравнение документов, анализ на соответствие и др.) могут получать данные от RAG и передавать результаты в Generator для включения в ответ.
- **Модуль парсинга и разметки → векторная БД:** обработанные документы (разбитые на блоки) и их эмбеддинги поступают в векторную БД для последующего поиска.
- **Модуль генерации отчётов:** получает данные от модуля RAG и/или агентов и формирует отчётный документ (DOCX) по шаблону.

Детализация потоков данных (форматы, протоколы) при необходимости уточняется в разделе 3 по каждому модулю и в документе «Описание информационного обеспечения».

## 3. Описание модулей (логика и функции)

### 3.1. Модуль пользовательского веб-интерфейса

**Назначение.** Модуль обеспечивает ввод запросов пользователя, загрузку документов, настройку параметров анализа и сравнения, отображение результатов и выгрузку отчётов. Пользовательский веб-интерфейс является основной точкой входа для работы с Системой (ТЗ п. 4.1.1). Описание даётся на уровне экранов и действий пользователя; логика не привязана к конкретному фреймворку (React, Vue.js или аналог).

**Состав и архитектура (SPA + BFF).** Модуль реализуется в виде одностраничного приложения (SPA) и сервера BFF (Backend for Frontend). BFF — минимальный сервер приложения, который: (1) раздаёт статические файлы SPA (HTML, JS, CSS); (2) отдаёт конфигурацию OIDC по запросу GET `/config.json` (discovery URL, client_id, redirect_uri и при необходимости базовый URL API), формируя её из переменных окружения, чтобы один и тот же билд SPA мог работать в разных окружениях (dev/prod) без изменений кода; (3) обрабатывает обмен кода авторизации на токен (POST `/auth/exchange`): принимает от SPA код, code_verifier и redirect_uri и выполняет обмен с IdP (token endpoint); (4) проксирует запросы к API: запросы SPA к тому же origin по пути `/api/*` BFF пересылает в API Gateway (Kong), подставляя JWT в заголовок `Authorization: Bearer <token>`. Браузер не обращается напрямую к IdP (кроме редиректа на страницу входа) и к Kong — только к BFF. Реализация BFF и сценарии входа описаны в документации IAM: iam/docs/frontend.md, iam/docs/auth-flow.md.

**Вход пользователя (OIDC, Authorization Code + PKCE).** Вход выполняется через единый вход (OIDC). (1) Пользователь открывает SPA по адресу BFF (например `http://localhost:3000/` или адрес развёртывания). (2) При необходимости входа SPA перенаправляет браузер на IdP (Authentik) с параметрами протокола Authorization Code + PKCE (code_challenge, state, redirect_uri, client_id, scope). (3) После успешной аутентификации IdP редиректит браузер на callback BFF (например `/callback?code=...&state=...`). (4) SPA передаёт код и code_verifier на BFF (POST `/auth/exchange`); BFF обменивает их на access_token (и при необходимости refresh_token) через token endpoint IdP. (5) Дальнейшие запросы к функциональности Системы (поиск, загрузка документов, отчёты и т.д.) SPA выполняет к тому же origin (BFF); BFF проксирует их в Kong с Bearer-токеном. Kong проверяет JWT (п. 3.8.1) и передаёт запрос на backend. Конфигурация (URL IdP, redirect_uri, client_id) задаётся на стороне BFF переменными окружения (OIDC_DISCOVERY_URL или AUTHENTIK_BASE_URL + OIDC_APP_SLUG, OIDC_REDIRECT_URI, OIDC_CLIENT_ID, KONG_INTERNAL_URL для проксирования) и не хранится в коде SPA. Значение redirect_uri должно совпадать с одним из Redirect URIs, настроенных у провайдера в Authentik; при несовпадении IdP возвращает ошибку invalid_grant (см. iam/docs/authentik.md).

**Основные сценарии (после аутентификации).** Инициация семантического поиска; загрузка документов (DOCX, PDF, изображения) для анализа или сравнения; настройка параметров (тип документа, объект сравнения, область поиска); просмотр результатов анализа, отчётов о различиях и рекомендаций; выбор шаблона отчёта и выгрузка результата в формате DOCX; управление составом документов в хранилище (для пользователей с соответствующими правами). Визуальный редактор разметки позволяет при необходимости корректировать границы логических блоков документа, сформированные парсером. Все эти действия выполняются через вызовы API (п. 3.8.1), проходящие через BFF с передачей JWT.

**Взаимодействие с backend и другими модулями.** Обмен данными с backend выполняется исключительно через вызовы API (REST over HTTPS). Цепочка запроса: браузер → BFF (тот же origin) → API Gateway (Kong) → backend-сервисы. BFF при проксировании подставляет в запрос JWT, полученный при обмене кода на токен; Kong проверяет JWT и передаёт контекст авторизации в backend (п. 3.8.1). Модуль веб-интерфейса не вызывает backend и IdP напрямую из браузера (кроме редиректа на страницу входа IdP). Развёртывание: в продакшене переменные окружения задаются на сервере BFF (или через секрет-менеджер); в Authentik у провайдера должен быть зарегистрирован redirect URI приложения (например `https://your-domain.com/callback`). Сравнение архитектур статического SPA и SPA + BFF с точки зрения безопасности — iam/docs/spa-vs-bff.md.

**Входы (логические):** действия пользователя (запросы, загружаемые файлы, параметры); при первом обращении — необходимость входа (редирект на IdP, затем код и code_verifier для BFF). **Выходы:** отображение результатов поиска, отчётов и сформированных документов; выгрузка файлов (DOCX); сообщения об ошибках и состояния загрузки.

### 3.2. Модуль RAG (Retriever и Generator)

Модуль RAG является ядром семантического поиска и генерации ответов в ИС «Фармадок» (ТЗ п. 4.1.1; документ «Описание архитектуры», раздел 6.4). Он объединяет извлечение релевантных фрагментов из локального хранилища документов (Retriever) и формирование ответа на основе этого контекста и при необходимости данных от агентов ИИ (Generator). Развёрнутое описание конвейера — подготовка данных, гибридный поиск (BM25 + векторный), слияние по RRF, переранжирование, слой рассуждений (reasoning) — приведено в документе **«Пояснительная записка к ТЗ. Система интеллектуального поиска документов»**; в настоящем пункте даётся описание логики модуля RAG и его связей с другими компонентами системы.

#### 3.2.1. Retriever (извлечение контекста)

Назначение Retriever — по запросу пользователя получить набор релевантных фрагментов документов с учётом прав доступа и передать их в Generator в качестве контекста. **(1)** Текст запроса преобразуется в векторное представление (эмбеддинг) с помощью модели эмбеддингов. **(2)** Выполняется поиск в векторной базе данных по метрике сходства (косинусное или аналог) с обязательной фильтрацией по RBAC: возвращаются только документы, доступные пользователю; контекст прав (идентификатор пользователя, роли) приходит от API Gateway (п. 3.8.1). События отказа в доступе к запрошенным данным логируются. **(3)** Выполняется отбор кандидатов (топ-k по релевантности). При реализации расширенного конвейера (см. документ по интеллектуальному поиску): лексический поиск (BM25), слияние результатов с векторным поиском по алгоритму RRF (Reciprocal Rank Fusion), при необходимости переранжирование с помощью Cross-Encoder для формирования финального списка кандидатов. **(4)** Отобранные фрагменты документов (контекст) с метаданными (источник, страница, раздел) передаются в Generator. **Взаимодействие:** с модулем векторной базы данных (п. 3.3) — запрос по эмбеддингу запроса и фильтрам доступа; с модулем аутентификации и авторизации — получение контекста прав через API Gateway (п. 3.8.2). Детальное описание алгоритма — раздел 6.1 настоящего документа.

#### 3.2.2. Generator (генерация ответа)

Назначение Generator — сформировать ответ пользователю на основе контекста от Retriever и при необходимости дополнительных данных от модуля агентов ИИ (п. 3.4). **(1)** Приём контекста (набор релевантных фрагментов документов) от Retriever и при необходимости структурированных данных от агентов (результаты сравнения документов, проверки соответствия, внешнего поиска и т.п.). **(2)** Маскирование персональных и чувствительных данных в контексте перед передачей в большую языковую модель — применяется Presidio или аналог (документ «Описание архитектуры», раздел 6.4). **(3)** Формирование промпта (запрос пользователя + контекст) и вызов большой языковой модели (БЯМ), размещённой локально на сервере организации или в доверенном облаке (ТЗ п. 4.1.1). **(4)** Обработка ответа модели: извлечение текста ответа и привязка утверждений к источникам (страницы, разделы документов). **(5)** Возврат сформированного ответа с обязательным цитированием первоисточников клиенту. Детальное описание алгоритма — раздел 6.2 настоящего документа.

#### 3.2.3. Конвейер и взаимодействие с другими модулями

Порядок выполнения: запрос пользователя поступает в модуль RAG → Retriever извлекает контекст из векторной БД с учётом RBAC → при наличии расширенного конвейера выполняется слой проверки качества контекста и/или переранжирование (см. документ «Пояснительная записка к ТЗ. Система интеллектуального поиска документов») → Generator формирует ответ на основе контекста и при необходимости данных от агентов → ответ возвращается клиенту. Модуль RAG получает от API Gateway контекст авторизации (идентификатор пользователя, роли) и передаёт его в Retriever для фильтрации документов по правам доступа; запрашивает векторную базу данных (п. 3.3) для семантического (и при гибридном поиске — лексического) поиска; при необходимости запрашивает дополнительные данные у модуля агентов (п. 3.4). Детали этапов оркестрации, гибридного поиска, RRF, Cross-Encoder, CRAG/Self-RAG приведены в документе «Пояснительная записка к ТЗ. Система интеллектуального поиска документов».

#### 3.2.4. Связь с критериями приёмки и алгоритмами

Реализация модуля RAG направлена на выполнение критериев приёмки системы (ТЗ п. 2.2): среднее время получения качественных результатов — не более 20 секунд; повышение качества анализа и оформления документов; обязательное цитирование источников в сгенерированных ответах. Ключевые алгоритмы семантического поиска и формирования ответа описаны в разделах 6.1 и 6.2 настоящего документа; полная детализация конвейера (подготовка данных, векторизация, гибридный поиск, переранжирование) — в документе «Пояснительная записка к ТЗ. Система интеллектуального поиска документов».

**Входы:** текст запроса пользователя; идентификатор пользователя или контекст прав от API Gateway (для RBAC в Retriever); при необходимости — параметры поиска (число кандидатов k, типы документов) и параметры генерации. **Выходы:** для Retriever — набор релевантных фрагментов документов с метаданными (источник, страница, раздел); для модуля в целом (Generator) — сгенерированный ответ с указанием источников (цитирование). Ключевые алгоритмы семантического поиска и формирования ответа описаны в разделе 6 настоящего документа; детализация конвейера интеллектуального поиска — в пояснительной записке по интеллектуальному поиску.

### 3.3. Модуль работы с векторной базой данных

Модуль обеспечивает хранение векторных представлений (эмбеддингов) документов и выполнение семантического поиска по сходству (k ближайших соседей, k-NN/ANN). Он является основой для работы Retriever в модуле RAG (п. 3.2). Реализация — Redis с расширениями векторного поиска или Pinecone, или аналог (ТЗ п. 4.1.1; документ «Описание архитектуры», раздел 6.5). Данные хранятся в зашифрованном виде; при индексации и при поиске соблюдаются требования к разграничению доступа (RBAC) и защите информации.

#### 3.3.1. Хранение и индексация

Назначение: принять эмбеддинги блоков документов и метаданные от конвейера индексации и сохранить их в векторной БД. Входные данные — векторные представления блоков (фрагментов) документов, сформированные моделью эмбеддингов после парсинга и при необходимости маскирования чувствительных данных (Presidio применяется до индексации — ТЗ п. 4.1.1, Architecture 6.5); метаданные включают идентификатор документа, тип документа, права доступа (для RBAC), при необходимости страницу и раздел. Обеспечивается логическое разделение по трём типам документов: **(1)** регламентирующие (правила, стандарты, фармакопейные статьи); **(2)** рабочие (инструкции по применению, ПУР, нормативные документы, изображения упаковки и т.п.); **(3)** кэш результатов внешнего поиска с ограничением времени хранения (TTL 24 часа — ТЗ, Architecture 6.5). Запись выполняется с учётом выбранного типа документа; при обновлении или удалении документа — соответствующее обновление или удаление записей в БД. Взаимодействие: конвейер индексации (модуль парсинга п. 3.7, модель эмбеддингов) передаёт данные в модуль векторной БД; маскирование персональных данных выполняется до передачи в конвейер индексации.

#### 3.3.2. Поиск по сходству

Назначение: по эмбеддингу запроса и параметрам поиска вернуть список релевантных фрагментов документов с учётом прав доступа. **(1)** Приём эмбеддинга запроса и параметров (число кандидатов k, при необходимости фильтры по типу документов) от Retriever (п. 3.2). **(2)** Контекст авторизации (идентификатор пользователя, роли), переданный вместе с запросом, используется для фильтрации — возвращаются только документы, доступные данному пользователю по RBAC. **(3)** Выполняется поиск k ближайших соседей (k-NN) или приближённый поиск (ANN) в соответствующем подмножестве БД (по типу документов и правам доступа). **(4)** Возврат списка идентификаторов документов/блоков и самих фрагментов текста (или ссылок на них) с метаданными (источник, страница, раздел) для передачи в Generator. Фильтрация по RBAC выполняется на уровне запроса к БД с передачей контекста прав от Retriever (п. 3.2.1). Взаимодействие: Retriever формирует эмбеддинг запроса и передаёт его в модуль векторной БД вместе с контекстом прав; модуль возвращает отфильтрованный список кандидатов.

#### 3.3.3. Защита данных: шифрование и управление ключами

Данные в векторной БД хранятся в зашифрованном виде (шифрование при хранении): применяется AES-256 или ГОСТ Р 34.12-2015 (ТЗ п. 4.1.1; документ «Описание архитектуры», разделы 6.5 и 8.3). Передача данных между компонентами (клиент векторной БД и сервисы) выполняется по TLS 1.3. Ключи шифрования хранятся в защищённом хранилище (HashiCorp Vault или аналог); ротация ключей — каждые 90 дней (ТЗ, Architecture). Модуль (или сервис, использующий векторную БД) при записи и чтении использует ключи из хранилища для шифрования и расшифрования. Конкретный механизм хранения и выдачи секретов описан в п. 3.8.2. Резервное копирование выполняется с шифрованием копий (ТЗ п. 4.1.3).

#### 3.3.4. Взаимодействие с другими модулями

Модуль векторной БД не инициирует запросы к другим компонентам; он предоставляет интерфейс для записи (индексация) и чтения (поиск). **Запись:** конвейер индексации (после парсинга документов — п. 3.5, построения эмбеддингов) передаёт в модуль векторной БД эмбеддинги и метаданные; маскирование персональных данных выполняется до индексации (Presidio). **Чтение:** Retriever модуля RAG (п. 3.2) передаёт эмбеддинг запроса и контекст прав; модуль векторной БД возвращает список кандидатов с учётом RBAC. Три типа документов (регламентирующие, рабочие, кэш внешнего поиска) позволяют при поиске ограничивать область запроса и применять политики хранения (в т.ч. TTL для кэша). Связь с критериями надёжности ТЗ (резервное копирование, время восстановления после сбоя) обеспечивается на уровне развёртывания; в описании ПО детали не приводятся.

**Входы:** при индексации — эмбеддинги блоков документов и метаданные (идентификатор документа, тип, права доступа, при необходимости страница/раздел); при поиске — эмбеддинг запроса, параметры (k, фильтры по типу документов), контекст прав для RBAC. **Выходы:** при индексации — подтверждение записи (успех или ошибка); при поиске — список идентификаторов и фрагментов документов (или ссылок на них) с метаданными для передачи в Retriever и далее в Generator.

### 3.4. Модуль агентов ИИ

Модуль агентов ИИ предоставляет расширяемый (плагинный) каркас для автономных агентов, выполняющих специализированные сценарии обработки документов и данных в изолированных, компонуемых единицах (ТЗ п. 4.1.1; документ «Описание архитектуры», раздел 6.6). Реализация строится на фреймворке оркестрации агентов LangChain, AutoGen или аналоге. Агенты запускаются в изолированных Docker-контейнерах с ограничением доступа к сети (белый список доменов), файловой системе и ресурсам; новые агенты подключаются через штатный интерфейс плагинов без изменения ядра системы (ТЗ п. 4.2.2). Результаты работы агентов при необходимости передаются в модуль RAG (Generator) или в модуль генерации отчётных документов (п. 3.2, 3.7).

#### 3.4.1. Оркестрация и поток выполнения задания

Оркестратор модуля агентов получает от вызывающего сервиса (backend) задание: тип операции и входные данные (документы, параметры проверки, запрос и т.п.). По типу операции выбирается соответствующий агент; задание передаётся агенту в изолированный контейнер. Агент выполняет последовательность вызовов: обращение к моделям (в т.ч. БЯМ при необходимости), парсерам документов, внутренним сервисам и при разрешённом сценарии — к внешним API из белого списка доменов. Результат (отчёт о сравнении, отчёт о несоответствиях, структурированная разметка, данные для RAG и т.д.) возвращается оркестратору. Оркестратор передаёт результат вызывающему сервису; при необходимости результат направляется в Generator модуля RAG (п. 3.2) в качестве дополнительного контекста или в модуль генерации отчётов (п. 3.6) для подстановки в шаблон DOCX. Контейнер агента по завершении задачи завершает работу; постоянное состояние вне векторной БД не сохраняется (Architecture 8.3). Взаимодействие: вызов от backend по запросу пользователя; передача данных в RAG (Generator) и в модуль генерации отчётов.

#### 3.4.2. Встроенные агенты первой очереди

В первой очереди реализации (ТЗ п. 4.2.1; Architecture 6.6, 9.1) в состав модуля входят следующие агенты. **Агент проверки соответствия (Compliance Agent):** проверка структуры, содержания и оформления инструкций по медицинскому применению (ИМП, SmPC/PIL) и иных документов на соответствие регуляторным требованиям (Минздрав, ЕЭК, ICH и т.п.) и референтным данным; формирование отчёта с указанием несоответствий. Входы: документ (или идентификатор), параметры проверки. Выходы: отчёт о несоответствиях. **Агент сравнения документов (Comparison Agent):** семантическое и формальное сравнение двух и более документов (инструкции, нормативные документы) с выявлением различий. Входы: два или более документа. Выходы: отчёт о различиях (семантических и формальных). **Агент разметки (Markup Agent):** разбиение документа на логические блоки (заголовки, разделы, таблицы) для последующей обработки БЯМ и индексации в векторной БД; может интегрироваться с визуальным редактором разметки. Входы: документ (DOCX, PDF). Выходы: структурированное представление (блоки с метаданными). Анонимизация персональных данных (Presidio) выполняется как middleware перед вызовом БЯМ и в описании архитектуры может относиться к отдельному компоненту; логически она обеспечивает маскирование ПД до передачи в любую БЯМ (в т.ч. при вызовах из агентов). Агенты второй очереди (веб-поиск, перевод, OCR) подключаются по мере реализации второй очереди (п. 3.7).

#### 3.4.3. Изоляция и проверка безопасности

Агенты выполняются в изолированных Docker-контейнерах («песочница») в соответствии с документом «Описание архитектуры», раздел 8.3, и ТЗ (п. 4.1.1, 4.2.2). **Сеть:** разрешён доступ только к заданному белому списку внутренних сервисов и внешних доменов; весь прочий исходящий трафик блокируется. **Файловая система:** монтирование только для чтения или в ограниченную область; доступа к файловой системе хоста и другим контейнерам нет. **Ресурсы:** ограничения по CPU и памяти задаются через механизмы контейнеризации (cgroups). **Жизненный цикл:** контейнер поднимается под задачу и уничтожается по завершении; постоянное состояние агента вне векторной БД не предусмотрено. **Проверка плагинов:** новые агенты, подключаемые как плагины (образы Docker), перед активацией проходят проверку безопасности — сканирование на вредоносный код (ClamAV или аналог) и при необходимости проверку на соответствие политикам (ТЗ п. 4.2.2; Architecture 6.6, 8.3). Детали развёртывания контейнеров и настройки сети приводятся в документе «Описание архитектуры и технических средств».

#### 3.4.4. Плагинный интерфейс и взаимодействие с RAG

Модуль поддерживает подключение новых агентов без изменения ядра системы через стандартный интерфейс плагинов: регистрация агента (образ Docker, метаданные), активация и деактивация, при необходимости обновление и удаление (Architecture 6.6, раздел 9). Взаимодействие с модулем RAG: агенты могут получать контекст из векторной БД (через интерфейс, предоставляемый системой) и передавать дополнительные данные в Generator (п. 3.2) для формирования ответа пользователю — например, результаты сравнения документов или проверки соответствия подставляются в итоговый ответ или в отчёт. Агенты второй очереди (веб-поиск, перевод, OCR) при их реализации подключаются через тот же плагинный интерфейс и выполняются в той же изолированной среде; их входы и выходы уточняются в документации по второй очереди (п. 3.7).

**Входы:** задание от вызывающего сервиса (тип операции, входные данные): для агента сравнения — два или более документа; для агента проверки соответствия — документ и параметры проверки; для агента разметки — документ (DOCX или PDF). **Выходы:** результат выполнения агента, передаваемый оркестратором вызывающему сервису и при необходимости в модуль RAG (Generator) или в модуль генерации отчётов: отчёт о различиях (сравнение), отчёт о несоответствиях (проверка соответствия), структурированное представление документа (разметка). Для агентов второй очереди форматы входов и выходов уточняются при их реализации.

### 3.5. Модуль парсинга и разметки документов

Модуль автоматически разбивает входящие документы на логические блоки (разделы, абзацы, таблицы и т.п.) для последующей обработки БЯМ и индексации в векторной БД (ТЗ п. 4.2.1, функция «Создание разметки документов»; документ «Описание архитектуры», раздел 6.7). Он является звеном конвейера индексации: результат парсинга и разметки передаётся на этап построения эмбеддингов и записи в векторную БД (п. 3.3). Поддерживаются форматы DOCX и PDF; при необходимости используется вывод OCR (вторая очередь). Используемые программные средства: python-docx, mammoth (DOCX), PyMuPDF (fitz), pdfplumber (PDF), LangChain text splitters или собственный сегментатор (раздел 4 настоящего документа).

#### 3.5.1. Парсинг документов (DOCX и PDF)

Назначение: извлечь из файла документа текст и структуру. **DOCX:** разбор формата OOXML — извлечение заголовков, абзацев, таблиц, списков, подписей к рисункам; при необходимости используется mammoth для сохранения сложной разметки. **PDF:** извлечение текстового слоя и позиционирования элементов (PyMuPDF/fitz, pdfplumber); учёт колонок, таблиц, сносок. Результат парсинга — плоский или иерархический список элементов с указанием типа (заголовок, раздел, подраздел, абзац, таблица, подпись к рисунку, сноска) и метаданных (уровень заголовка, порядок следования). При ошибках парсинга (повреждённый файл, неподдерживаемая структура) модуль возвращает вызывающему сервису код или описание ошибки.

#### 3.5.2. Сегментация и формирование блоков для индексации

Назначение: преобразовать результат парсинга в список логических блоков (chunks), пригодных для построения эмбеддингов и записи в векторную БД. Правила сегментации: объединение содержимого по заголовкам и разделам, ограничение размера блока (например, по числу токенов), сохранение целостности таблиц и списков. Для каждого блока формируются метаданные: тип блока, иерархия (уровень заголовка, родительский раздел), координаты в исходном документе (страница, диапазон), при необходимости идентификатор документа. Используются LangChain text splitters или собственный сегментатор (раздел 4). Результат передаётся конвейеру индексации (п. 3.3) и при необходимости агенту разметки (п. 3.4) или визуальному редактору разметки.

#### 3.5.3. Визуальный редактор разметки

В веб-интерфейсе для уполномоченных пользователей предоставляется визуальный редактор разметки (документ «Описание архитектуры», раздел 6.7; ТЗ п. 4.2.1 — «возможность ручного редактирования разметки через визуальный интерфейс»). Назначение: отображение автоматически определённых границ блоков и их ручная корректировка (объединение или разбиение блоков, уточнение границ). Сохранённая после правок разметка используется конвейером индексации при построении эмбеддингов. Модуль парсинга и разметки предоставляет API для загрузки документа, получения текущей разметки и сохранения отредактированной разметки; реализация экрана редактора входит в модуль пользовательского веб-интерфейса (п. 3.1).

#### 3.5.4. Взаимодействие с другими модулями

Модуль получает на вход документ (файл DOCX или PDF) от пользователя или от конвейера загрузки; при реализации второй очереди — распознанный текст или блоки от модуля OCR. Результат работы — структурированное представление документа и список блоков с координатами и метаданными — передаётся конвейеру индексации (п. 3.3) для построения эмбеддингов и записи в векторную БД. Агент разметки (п. 3.4) может вызывать функции парсинга и сегментации или использовать общий результат модуля. Визуальный редактор разметки (п. 3.1) запрашивает и сохраняет разметку через API backend. Чувствительные данные в тексте блоков маскируются перед индексацией (Presidio); место маскирования в конвейере (до или после сегментации) уточняется в документе «Описание архитектуры».

**Входы:** документ в формате DOCX или PDF; при реализации второй очереди — распознанный текст или блоки от модуля OCR; при сохранении правок — отредактированная разметка от веб-интерфейса. **Выходы:** структурированное представление документа (элементы с типами и иерархией); список логических блоков с координатами (страница, диапазон) и метаданными (тип блока, иерархия, идентификатор документа) для конвейера индексации и визуального редактора; при ошибке парсинга — код или описание ошибки для вызывающего сервиса.

### 3.6. Модуль генерации отчётных документов

Модуль формирует структурированные отчётные документы в формате DOCX на основе предопределённых шаблонов и данных, поступающих от модуля RAG и агентов ИИ (ТЗ п. 4.2.1, функция «Генерация отчётных документов на основе типовых шаблонов»; документ «Описание архитектуры», раздел 6.10). Шаблоны предоставляются Заказчиком или разрабатываются Подрядчиком; поддерживается несколько шаблонов, выбор при формировании отчёта выполняют администратор или уполномоченные пользователи. Реализация опирается на библиотеку python-docx или аналог (раздел 4 настоящего документа).

#### 3.6.1. Шаблоны отчётов и их выбор

Назначение: обеспечить использование типовых шаблонов и выбор нужного шаблона при формировании отчёта. Шаблоны представляют собой файлы DOCX с размеченными областями подстановки (плейсхолдеры, поля, при необходимости таблицы и повторяющиеся секции). Управление шаблонами: загрузка, хранение и при необходимости версионирование; администраторы загружают и управляют файлами шаблонов через веб-интерфейс (Architecture 6.10; ТЗ — возможность выбора шаблона администратором и пользователями). При запросе на генерацию отчёта пользователь или администратор выбирает шаблон; при наличии единственного шаблона по умолчанию выбор может быть неявным. Модуль получает идентификатор выбранного шаблона и набор данных для подстановки.

#### 3.6.2. Данные для подстановки и логика формирования отчёта

Назначение: подставить в шаблон переданные данные и сформировать итоговый документ DOCX. Входные данные — результаты анализа на соответствие регуляторным требованиям (отчёт о несоответствиях от агента проверки), сравнения документов (отчёт о различиях от агента сравнения), семантического поиска (ответ RAG с цитированием источников), при необходимости дополнительные структурированные сведения от модуля RAG (Generator) и агентов (п. 3.3, 3.5). Логика: загрузка выбранного шаблона DOCX; сопоставление полей данных с плейсхолдерами и областями шаблона по правилам, заданным в шаблоне (имена полей, структура таблиц, повторяющиеся блоки); подстановка текста и структурированных фрагментов; сохранение форматирования, структуры и требований к оформлению Заказчика (Architecture 6.10). При отсутствии значения для обязательного поля выполняется подстановка пустой строки или маркера согласно правилам шаблона; при ошибке (например, несовместимая структура данных) модуль возвращает вызывающему сервису код или описание ошибки.

#### 3.6.3. Вывод отчёта и сохранение

Сформированный документ сохраняется в формате DOCX. Далее возможны возврат файла клиенту для скачивания через веб-интерфейс и/или сохранение экземпляра отчёта в хранилище системы в соответствии с правилами развёртывания (ТЗ — «сохранение результатов анализа в формате DOCX по шаблону»). Права на генерацию отчётов определяются ролевой моделью (п. 3.8.1): администратор и эксперты имеют полный доступ к генерации отчётов; аналитик может формировать отчёты без права изменять состав документов в хранилище (документ «Описание архитектуры», раздел 8.1). Факт генерации отчёта (пользователь, выбранный шаблон, время) логируется для целей аудита.

#### 3.6.4. Взаимодействие с другими модулями

Модуль вызывается backend по запросу пользователя после выполнения анализа, сравнения или поиска. Данные для подстановки поступают от модуля RAG (результаты семантического поиска, сгенерированный ответ с цитированием) и от модуля агентов ИИ (отчёты проверки соответствия, отчёты сравнения документов) (п. 3.3, 3.5). Шаблоны хранятся в хранилище, доступном модулю; управление шаблонами (загрузка, удаление) выполняется через веб-интерфейс с правами администратора (п. 3.1). Связь с разделом 5 настоящего документа: сформированный файл DOCX входит в сводную таблицу выходных данных системы.

**Входы:** идентификатор выбранного шаблона отчёта (или файл шаблона DOCX при прямой передаче); набор данных для подстановки — результаты проверки на соответствие, сравнения документов, семантического поиска (ответ RAG), при необходимости дополнительные структурированные данные от агентов. **Выходы:** сформированный файл отчёта в формате DOCX (передача клиенту для скачивания и/или сохранение в хранилище); при ошибке — код или описание ошибки для вызывающего сервиса.

### 3.7. Модули второй очереди (кратко)

Функции второй очереди реализации заданы ТЗ п. 4.2.2 и документом «Описание архитектуры», раздел 9.2. Ниже приведено краткое описание модулей и подсистем второй очереди; детальная логика, входы и выходы уточняются при техническом проектировании и реализации второй очереди.

#### 3.7.1. Безопасный веб-поиск

Модуль (агент) обеспечивает контролируемый поиск по разрешённым внешним источникам (белый список доменов, настраиваемый администратором: EMA, FDA, WHO, Росздравнадзор и т.п.). Запросы, содержащие конфиденциальную информацию, не передаются вовне и обрабатываются только по внутреннему корпусу документов (фильтр конфиденциальности). Получение данных — через REST API или веб-скрейпинг в зависимости от ресурса. Результаты кэшируются в векторной БД (TTL 24 часа) и доступны для семантического поиска вместе с внутренними документами; интеграция с модулем RAG (документ «Описание архитектуры», раздел 6.11; ТЗ п. 4.2.2). Поддержка доступа к патентным и международным фармацевтическим реестрам; при необходимости — визуализация перевода проблем безопасности (риски, побочные действия) в интерфейсе.

#### 3.7.2. Перевод документов

Модуль перевода обеспечивает автоматический перевод фармацевтических документов с иностранных языков (английский, немецкий, французский и др.) на русский с учётом терминологии и регуляторных требований. Используется локальная или доверенная облачная БЯМ (ТЗ п. 4.2.2; документ «Описание архитектуры», раздел 6.9). Сохраняется структура исходного документа (разделы, таблицы, заголовки); выход — документ в формате DOCX с сохранением оформления. Поддерживается сценарий двуязычного сравнения ПУР (русская и английская версии) (документ «Описание архитектуры», раздел 9.2).

#### 3.7.3. Оптическое распознавание текста (OCR)

Модуль (агент) извлекает машиночитаемый текст из отсканированных документов, изображений и PDF без текстового слоя. Входные форматы: PDF (на основе изображений), JPEG, PNG. Распознавание выполняется для фармацевтических документов (упаковка, маркировка, отсканированные регуляторные формы). Выход — обычный текст и структурированные блоки для передачи в модуль парсинга и разметки (п. 3.5); распознанный текст индексируется в векторной БД для семантического поиска (документ «Описание архитектуры», раздел 6.8; ТЗ п. 4.2.2). Технологии: Tesseract OCR, PaddleOCR или аналог; реализация — в рамках модуля агентов (п. 3.4).

#### 3.7.4. Управление плагинами агентов ИИ

Подсистема предоставляет стандартный интерфейс регистрации, проверки и управления жизненным циклом агентов-плагинов без изменения ядра системы (ТЗ п. 4.2.2; документ «Описание архитектуры», раздел 6.12). Регистрация: приём новых агентов в виде образов Docker через API. Проверка безопасности перед активацией: статический анализ, сканирование на вредоносный код (ClamAV или аналог), при необходимости проверка на соответствие политикам. Управление жизненным циклом: активация, деактивация, обновление, удаление плагинов. Каждый плагин выполняется в отдельном контейнере с заданными ограничениями по ресурсам и сети. Каталог плагинов доступен администраторам через веб-интерфейс (документ «Описание архитектуры», разделы 6.12 и 9.2). Детали API и сценариев уточняются при реализации второй очереди.

Дополнительные сценарии второй очереди (двуязычное сравнение ПУР, сравнение ИИ-резюме ПУР с оригиналом, кросс-языковой поиск по внутренним и международным источникам) могут опираться на перечисленные выше модули; их описание приводится в документации по второй очереди и в документе «Описание архитектуры», раздел 9.2.

Форматы входов и выходов по каждому модулю второй очереди фиксируются при техническом проектировании и реализации (см. раздел 9 документа «Описание архитектуры» и последующие версии настоящего документа).

### 3.8. Безопасность

#### 3.8.1. Подсистема входа, аутентификации и авторизации

##### 3.8.1.1. Обзор подсистемы

Подсистема входа, аутентификации и авторизации обеспечивает единый вход для всех клиентских запросов к серверной части, установление личности пользователя (аутентификация) и проверку прав доступа (авторизация по RBAC). Состав: **API Gateway** (Kong) — единственная точка входа, проверка JWT и RBAC, ограничение частоты, маршрутизация на backend-сервисы; **IdP** (Authentik) — аутентификация, выдача JWT с утверждениями о ролях; **BFF** — обмен кода на токен и проксирование запросов к API с Bearer-токеном; **SPA** — браузерное приложение. Поток: запросы к backend проходят только через шлюз; вход пользователя — через IdP (OAuth2/OIDC, Authorization Code + PKCE); при последующих запросах шлюз проверяет JWT и права, передаёт запрос на backend с контекстом (идентификатор пользователя, роли). Детализация компонентов — в подпунктах «API Gateway и точка входа» и «Аутентификация и авторизация» ниже.

##### 3.8.1.2. API Gateway и точка входа

**Назначение.** Модуль API Gateway является единственной точкой входа для всех клиентских запросов к серверной части системы (веб-интерфейс через BFF, внешние клиенты, мобильные приложения). Все запросы к backend-сервисам проходят через шлюз; прямой доступ к backend в обход шлюза не предусмотрен. Шлюз обеспечивает принудительную аутентификацию (проверка JWT), проверку прав доступа (RBAC), ограничение частоты запросов (не более 100 запросов в минуту на пользователя — ТЗ п. 4.1.1), маршрутизацию трафика на соответствующие сервисы и аудит всех запросов. Выбор обусловлен требованиями ТЗ п. 4.1.1 (единая точка входа, аутентификация, RBAC, ограничение частоты, логирование) и п. 4.1.4 (контроль доступа, журналирование).

**Реализация.** В конфигурации на стадии прототипирования используется Kong OSS (документ «Описание архитектуры», раздел 6.2; ТЗ п. 4.1.1). Конфигурация шлюза задаётся декларативно: файл `iam/kong/kong.yml` описывает маршруты, плагины и параметры; Kong работает в режиме DB-less. Внешний доступ к API — по порту 8001 (HTTP); при необходимости добавляется HTTPS (порт 8444) и reverse proxy с TLS. Маршрут `/api` проксируется на backend-сервисы. Подробнее — iam/docs/kong.md.

**Конфигурация на стадии прототипирования и перспектива замены.** Kong выбран в качестве API Gateway на стадии прототипирования. Выявлены недостатки: усложнённая схема авторизации (в Kong OSS отсутствует плагин openid-connect; проверка RBAC по ролям из JWT требует плагина pre-function и кода на Lua). На этапе технической реализации будет рассмотрена возможность замены Kong на **Apache APISIX** (плагин openid-connect, упрощение настройки). Архитектура цепочки (BFF → API Gateway → backend, проверка JWT и RBAC на шлюзе) при замене сохраняется.

**Проверка JWT (аутентификация).** Шлюз извлекает токен из заголовка `Authorization: Bearer <token>` и проверяет его по ключам IdP (JWKS/PEM, при необходимости issuer). В реализации используется плагин **jwt** с подстановкой ключей из IdP Authentik (скрипт `iam/kong/scripts/setup-kong-jwt-auth.sh`). При отсутствии или невалидном токене — **401 Unauthorized**, запрос на backend не передаётся.

**Последовательность обработки запроса.** (1) Приём запроса и извлечение JWT. (2) Аутентификация: проверка подписи и срока действия JWT по ключам IdP. (3) Авторизация (RBAC): для пары «метод HTTP + путь» проверка разрешённости операции для ролей из JWT; при недостаточных правах — **403 Forbidden**. (4) Ограничение частоты: не более 100 запросов в минуту на пользователя; при превышении — **429 Too Many Requests**. (5) Маршрутизация: запрос с контекстом авторизации направляется на backend-сервис; при таймауте или ошибке — ответ 5xx. Логирование каждого запроса (см. абзац «Логирование и взаимодействие» в настоящем подпункте).

**Ограничение частоты и защита (WAF).** Ограничение — не более 100 запросов в минуту на пользователя (ТЗ п. 4.1.1). Защита от DDoS — интеграция с WAF организации.

**Маршрутизация и передача контекста.** По пути и методу HTTP запрос направляется на соответствующий backend-сервис; контекст авторизации (идентификатор пользователя, роли) формируется на основе JWT и правил RBAC (см. п. 3.8.1). Backend использует контекст для проверки прав при доступе к данным.

**Логирование и взаимодействие.** Все запросы логируются с метаданными (IP, идентификатор пользователя, время, метод и путь, статус ответа, задержка). События 401, 403, 429 фиксируются отдельно и могут инициировать оповещения в SIEM. Журналы передаются в систему SIEM Заказчика. С модулем аутентификации (п. 3.8.1): проверка JWT по ключам IdP, контекст прав для RBAC и backend. С backend: передача запроса и контекста, приём ответа.

**Входы:** HTTP-запрос (метод, путь, заголовки, тело), JWT в заголовке авторизации. **Выходы:** вызов backend с передачей запроса и контекста авторизации и возврат ответа клиенту либо HTTP-ответ при ошибке (401, 403, 429, 5xx); запись в журнал.

##### 3.8.1.3. Аутентификация и авторизация

**Назначение и состав модуля.** Модуль аутентификации и авторизации выступает источником идентичности пользователя и сведений о его правах для всей системы. Он обеспечивает: установление личности пользователя при входе (аутентификация), выдачу токенов доступа (JWT) для последующих запросов к API, передачу контекста прав (роли, разрешённые операции) в API Gateway и backend-сервисы для принятия решений о доступе (авторизация). Требования ТЗ: п. 4.1.1 (единый вход, разграничение доступа), п. 4.1.4 (защита информации, MFA при необходимости). Модуль реализуется как совокупность взаимодействующих компонентов: (1) **IdP (Identity Provider)** — в реализации на стадии прототипирования Authentik (или Keycloak по согласованию с Заказчиком); (2) **BFF** — через него браузерный клиент (SPA) входит в систему и обращается к API; (3) **SPA** — одностраничное приложение в браузере. Логически модуль располагается между клиентом и API Gateway; backend-сервисы получают контекст прав от шлюза и не обращаются к IdP напрямую. *Входы и выходы:* учётные данные при входе; код авторизации и code_verifier при обмене на токен; JWT при последующих вызовах API. *Выходы:* JWT после успешной аутентификации; контекст прав для API Gateway и backend; HTTP-ответы при ошибках. Детализация потока и BFF — в следующих абзацах; хранение секретов IdP — п. 3.8.2.

**Реализация IdP (Authentik): настройка и интеграция.** В конфигурации на стадии прототипирования используется Authentik 2026.2. Секреты IdP подставляются через скрипт iam/authentik/scripts/run-authentik-with-vault.sh и хранилище Vault (детали — п. 3.8.2; iam/docs/vault.md). В Authentik создаётся провайдер OpenID Connect и приложение; у провайдера задаются Redirect URIs (совпадение с BFF), подписывающий ключ, сроки действия токенов, scopes; для RBAC настраивается включение ролей/групп в JWT. JWKS используется Kong для проверки JWT; подстановка ключей — скрипт iam/kong/scripts/setup-kong-jwt-auth.sh. Переменные окружения BFF и сценарии развёртывания — iam/docs/frontend.md, iam/docs/authentik.md.

**Процесс аутентификации и конфигурация BFF.** Участники: браузер (SPA), BFF, IdP (Authentik), API Gateway (Kong), backend. Пошаговая последовательность: (1) пользователь открывает SPA по адресу BFF; (2) при необходимости входа SPA перенаправляет на IdP (Authorization Code + PKCE); (3) после успешной аутентификации IdP редиректит на callback BFF с кодом; (4) SPA отправляет на BFF POST `/auth/exchange` (code, code_verifier, redirect_uri); BFF обменивает с IdP и возвращает access_token; (5) при последующих запросах BFF подставляет JWT и проксирует в Kong; (6) Kong проверяет JWT и авторизацию (п. 3.8.1); при успехе передаёт запрос на backend с контекстом прав. Конфигурация BFF: OIDC_DISCOVERY_URL (или AUTHENTIK_BASE_URL + OIDC_APP_SLUG), OIDC_REDIRECT_URI, OIDC_CLIENT_ID, KONG_INTERNAL_URL. Redirect URI должен совпадать с настройками провайдера в Authentik. Роли/группы в JWT настраиваются в IdP для RBAC в Kong и backend.

**Ролевая модель и проверка авторизации (RBAC).** Назначение ролей — средствами IdP; матрица ролей уточняется на этапе технического проектирования. Предварительные роли (документ «Описание архитектуры», раздел 8.1): Администратор, Эксперт, Аналитик, Пользователь с доступом только на чтение. Проверка авторизации: на уровне Kong — для пары «метод HTTP + путь» (при недостаточных правах — 403); на уровне backend — при доступе к данным и при изменяющих операциях. Retriever (п. 3.2) использует контекст прав для фильтрации документов по RBAC. Настройка: в IdP — выдача ролей/групп в JWT (имя claim, например `groups`, `roles`); в Kong — правила «метод + путь → разрешённые роли» (в реализации на стадии прототипирования — плагин pre-function, таблица правил в конфиге). Порядок плагинов: jwt, затем проверка RBAC. Подробнее — iam/docs/kong.md.

**Взаимодействие с другими компонентами.** BFF и IdP: обмен кода на токен (POST на token endpoint IdP). BFF и Kong: проксирование запросов с Bearer-токеном. Kong и IdP: проверка JWT по ключам IdP (JWKS/PEM), без вызова IdP при каждом запросе. Kong и backend: передача запроса и контекста авторизации; сервисы (в т.ч. Retriever, п. 3.3) используют контекст для RBAC. События 401, 403 фиксируются для аудита и SIEM (п. 3.2, 3.3).

#### 3.8.2. Управление ключами и секретами

Назначение: хранение секретов (пароли БД, секретные ключи приложений, при необходимости ключи шифрования) в защищённом хранилище — HashiCorp Vault или аналог. В реализации на стадии прототипирования механизм приведён на примере IdP Authentik: в Vault хранятся пароль PostgreSQL для Authentik и секретный ключ IdP; используется хранилище типа KV v2 с отдельным путём и ключами (например `postgres_password`, `secret_key`).

**Запуск и режимы.** Vault поднимается отдельным сервисом (например, в контейнере); доступ по API и при необходимости по веб-интерфейсу (порт 8200). В режиме разработки допускается одноузловая конфигурация с хранилищем в памяти и одним root-токеном; для производства предусматривается production mode с персистентным хранилищем и процедурой unseal. Хранилище типа KV v2 с отдельным путём для секретов каждого сервиса.

**Получение кредов при старте.** Принят подход «подготовка кредов перед запуском стека». Перед запуском сервисов выполняется скрипт на хосте или в среде CI, который по токену Vault запрашивает нужные секреты и записывает их во временный файл (например `.env.vault`); контейнеры подключают этот файл через `env_file` и не получают токен Vault. Преимущества: минимальные привилегии, токен Vault не попадает в контейнеры, скомпрометированный контейнер не может запрашивать секреты других сервисов. Альтернатива — запрос кредов при старте каждого контейнера (entrypoint) — требует передачи токена в контейнеры; при Docker Compose предпочтительна подготовка до запуска.

**Дополняемый список приложений и ключей в Vault.** Секреты хранятся в KV v2 по путям вида `secret/farmadoc/<сервис>`. Состав уточняется по ходу разработки технического проекта.

| Сервис / приложение | Путь в Vault (пример) | Ключи (назначение) |
|---------------------|------------------------|----------------------|
| Authentik (IdP) | `secret/farmadoc/authentik` | `postgres_password` — пароль БД IdP; `secret_key` — секретный ключ приложения (AUTHENTIK_SECRET_KEY). |
| API Gateway (Kong) | `secret/farmadoc/kong` | Ключи и пароли по мере настройки (например, для плагинов, админ-API). |
| Backend-сервисы | `secret/farmadoc/backend` | Секреты подключения к БД, API-ключи, внутренние токены — по мере введения. |
| Векторная БД / шифрование | `secret/farmadoc/vector-db` | Ключи шифрования данных при хранении (при использовании такого режима). |
| *(резерв для иных сервисов)* | `secret/farmadoc/<сервис>` | Уточняется при подключении сервиса. |

**Ротация секретов: ручная и автоматическая.** *Ручная:* (1) обновить значение секрета в Vault (через API, CLI или UI); (2) скриптом обновить файл с кредами из Vault без перезапуска сервисов (режим «только запись файла»); (3) перезапустить затронутые контейнеры. Для ротации пароля БД: сначала сменить пароль в самой СУБД (например, `ALTER USER ... PASSWORD '...'`), затем обновить значение в Vault и выполнить шаги (2)–(3); иначе приложение получит новый пароль из Vault, а в БД останется старый — подключение будет падать. Ручная ротация применяется при разовых сменах (компрометация, смена политики, вывод сотрудника). *Автоматическая:* периодичность (в т.ч. каждые 90 дней для ключей шифрования по п. 4.1.1 ТЗ) задаётся политикой при развёртывании. Варианты: по расписанию (cron / systemd timer) — задача генерирует или получает новый секрет, записывает в Vault, обновляет файл кредов и перезапускает затронутые сервисы; для паролей БД — предварительная смена в СУБД; динамические секреты Vault (движки для БД и т.п.) с TTL; интеграция с CI/CD — ротация в пайплайне развёртывания. В конфигурации репозитория на стадии прототипирования автоматическое расписание может не быть предустановлено; его вводят при развёртывании.

*Примечание.* Секреты иных сервисов системы (API Gateway, модули backend, при необходимости ключи шифрования векторной БД и т.д.) хранятся по тому же принципу: отдельный путь в хранилище, подготовка кредов перед стартом или по процедуре ротации, без передачи токена Vault в рабочие контейнеры.

*Примечание.* Описание механизма будет уточняться по ходу разработки технического проекта (подключение дополнительных сервисов, политики ротации, интеграция с CI/CD). Подробнее — iam/docs/vault.md.

#### 3.8.3. Безопасность данных и маскирование персональных данных

Все соединения между компонентами системы защищаются протоколом TLS 1.3. Шифрование данных при хранении и функции хэширования выполняются по ГОСТ Р 34.12-2015 и ГОСТ Р 34.11-2012 в соответствии с п. 4.1.4 ТЗ. Персональные и иные чувствительные данные маскируются перед передачей в большую языковую модель (Generator, п. 3.2.2) и перед индексацией в векторной БД (п. 3.3.1) с использованием Microsoft Presidio или функционального аналога (п. 4.1.1, п. 4.1.4 ТЗ). Загружаемые плагины агентов (вторая очередь) перед активацией проверяются на вредоносный код средствами ClamAV или аналога (п. 4.2.2 ТЗ; см. также п. 3.4.3, 3.7.4). Журналы событий и действий передаются в систему SIEM Заказчика (по протоколу syslog или через API) в соответствии с п. 4.1.4 ТЗ; логирование запросов через API Gateway — п. 3.8.1. Детали шифрования при хранении в векторной БД и управления ключами — п. 3.3.3, 3.8.2.

## 4. Используемые программные средства (библиотеки и фреймворки)

В настоящем разделе перечислены основные программные средства (библиотеки, фреймворки и внешние компоненты), применяемые при реализации модулей системы, описанных в разделе 3. Указание конкретных наименований и версий необходимо для однозначной идентификации зависимостей при сборке, сопровождении и валидации ПО в соответствии с практикой ЕСПД (ГОСТ 19 серии). Описание даётся в разрезе серверной части приложения, клиентской части (веб-интерфейс) и внешних компонентов, используемых системой, но развёртываемых и настраиваемых отдельно. Детали установки, конфигурирования и развёртывания перечисленных средств относятся к документу «Описание архитектуры и технических средств»; полный перечень технологий и версий приведён в документе «Описание архитектуры» (Architecture.md), раздел 7.

### 4.1. Серверная часть

- **Язык программирования:** Python 3.11 и выше (backend-сервисы и ИИ-пайплайны).
- **Фреймворк API:** FastAPI или функциональный аналог (реализация REST API, маршрутизация, валидация запросов).
- **Фреймворки агентов ИИ:** LangChain, AutoGen или аналоги (оркестрация агентов, цепочки вызовов, интеграция с моделями).
- **Работа с документами:** python-docx (генерация и парсинг DOCX), mammoth (парсинг DOCX), PyMuPDF (fitz), pdfplumber (парсинг PDF).
- **Эмбеддинги и векторный поиск:** библиотеки для вызова моделей эмбеддингов и клиенты векторной БД (в соответствии с выбранным стеком, см. документ «Описание архитектуры», раздел 7).
- **Разбиение текста на блоки:** LangChain text splitters или собственный сегментатор.

Назначение перечисленных средств в логике программы: обеспечение REST API, оркестрация агентов, преобразование документов в структурированный вид и в эмбеддинги, запись и поиск в векторной БД, формирование отчётных файлов.

### 4.2. Клиентская часть

- **Фреймворк SPA:** React или Vue.js (по выбору при разработке). Назначение: реализация пользовательского интерфейса, отправка запросов к API, отображение результатов и управление состоянием экранов.

### 4.3. Внешние компоненты

- **Authentik (IdP)** — аутентификация и авторизация (SSO, MFA, OAuth2/OIDC, выдача JWT); или Keycloak по согласованию с Заказчиком. Документация по настройке: iam/docs/authentik.md, iam/docs/auth-flow.md.
- **API Gateway (Kong или аналог)** — единая точка входа, проверка JWT и RBAC, ограничение частоты запросов, маршрутизация (компонент может быть внешним по отношению к коду приложения).
- **Векторная база данных (Redis/Pinecone или аналог)** — хранение эмбеддингов и поиск по сходству.
- **HashiCorp Vault** — хранение секретов и при необходимости ключей шифрования; применение в системе (в т.ч. на примере Authentik) описано в п. 3.8.2.
- **Microsoft Presidio или аналог** — маскирование персональных и чувствительных данных перед передачей в БЯМ и перед индексацией в векторной БД (подробнее — п. 3.8.3).

Роль перечисленных компонентов в логике ПО: обеспечение идентификации, безопасности, хранения векторных данных и защиты персональных данных. Вопросы установки и развёртывания этих компонентов относятся к документу «Описание архитектуры и технических средств».

Полный перечень технологий приведён в документе «Описание архитектуры» (Architecture.md), раздел 7.

## 6. Алгоритмы (ключевые)

### 6.1. Алгоритм семантического поиска

1. Получение текста запроса и идентификатора пользователя.
2. Преобразование запроса в векторное представление (эмбеддинг) с помощью модели эмбеддингов.
3. Поиск в векторной базе данных по метрике сходства (например, косинусное) с учётом прав доступа (RBAC): возвращаются только документы, доступные пользователю. При отказе в доступе к запрошенным данным — логирование события.
4. Отбор кандидатов (топ-k по релевантности). При использовании гибридного поиска (лексический + векторный) — слияние результатов (например, по алгоритму RRF) и при необходимости переранжирование (Cross-Encoder).
5. Передача отобранных фрагментов документов (контекста) в компонент Generator для формирования ответа.

Детализация конвейера (в т.ч. гибридный поиск, переранжирование) приведена в документе «Пояснительная записка к ТЗ. Система интеллектуального поиска документов».

### 6.2. Алгоритм формирования ответа (RAG)

1. Получение контекста от Retriever (набор релевантных фрагментов документов) и при необходимости дополнительных данных от агентов ИИ.
2. Проверка и при необходимости маскирование персональных и чувствительных данных в контексте (Presidio или аналог) перед передачей в БЯМ.
3. Формирование промпта для большой языковой модели (запрос пользователя + контекст) и вызов БЯМ.
4. Обработка ответа модели: извлечение текста ответа и привязка к источникам (страницы, разделы документов).
5. Возврат сформированного ответа с цитированием источников клиенту.

### 6.3. Алгоритм индексации документа

1. Получение документа (файл DOCX, PDF или результат OCR) и метаданных (тип документа, права доступа и т.п.).
2. Парсинг документа и разбиение на логические блоки (модуль парсинга и разметки).
3. При необходимости маскирование чувствительных данных в тексте блоков.
4. Преобразование каждого блока (или объединённых блоков) в векторное представление (эмбеддинг) с помощью модели эмбеддингов.
5. Запись эмбеддингов и метаданных в векторную базу данных с учётом логического разделения по типам документов (регламентирующие, рабочие, кэш внешнего поиска) и разграничения доступа.

Блок-схемы указанных алгоритмов при необходимости добавляются в приложения к настоящему документу или в сопроводительные материалы по валидации ПО.
