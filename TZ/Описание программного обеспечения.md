## 1. Введение

### 1.1. Назначение программы

Настоящий документ описывает программное обеспечение информационной системы «Фармадок», предназначенной для автоматизации поиска, обработки и анализа фармацевтической документации с использованием технологий искусственного интеллекта на основе больших языковых моделей (см. раздел 2 Технического задания на создание ИС «Фармадок»). Система ориентирована на использование в ФГБУ «НЦЭСМП» Минздрава России, а также в фармацевтических компаниях, регуляторных органах и организациях, занимающихся оценкой соответствия и контролем качества лекарственных средств.

Описание выполнено в соответствии с ГОСТ 19.402-78 и ЕСПД (ГОСТ 19 серии) и предназначено для разработчиков (доработка и сопровождение кода) и инженеров по валидации ПО.

### 1.2. Условные обозначения и сокращения

Условные обозначения и сокращения приведены в разделе 1 Технического задания на создание ИС «Фармадок». Основные: БЯМ (большая языковая модель), RAG (Retrieval-Augmented Generation), RBAC (Role-Based Access Control), API (Application Programming Interface), DOCX (формат файлов Microsoft Word), MFA (Multi-Factor Authentication), НЦЭСМП (Научный центр экспертизы средств медицинского применения), ПУР (план управления рисками), TLS (Transport Layer Security), WAF (Web Application Firewall).

## 2. Логическая структура программы

### 2.1. Общая структура

Программный комплекс ИС «Фармадок» представляет собой совокупность логических модулей (компонентов), взаимодействующих по заданным правилам. Описание даётся без привязки к физическим серверам — только логические модули и потоки данных между ними.

**Основные логические компоненты:**

- Клиентская часть (пользовательский веб-интерфейс) — одностраничное приложение, выполняемое в браузере.
- API Gateway — единая точка входа для всех запросов от клиента к серверной части.
- Сервисы backend: аутентификация и авторизация, модуль RAG (Retriever и Generator), модуль агентов ИИ, модуль парсинга и разметки документов, модуль генерации отчётных документов.
- Подсистема работы с векторной базой данных (хранение и поиск эмбеддингов документов).
- Взаимодействие с большой языковой моделью (БЯМ) для генерации ответов.

Состав модулей согласован с документом «Описание архитектуры» (Architecture.md), раздел 6. Детальное описание каждого модуля приведено в разделе 3 настоящего документа.

### 2.2. Взаимодействие модулей

Потоки управления и данных между модулями организуются следующим образом.

- **Клиент → API Gateway:** все запросы пользователя (поиск, загрузка документов, запуск анализа/сравнения, выгрузка отчётов) поступают в систему через API Gateway по протоколу HTTPS.
- **API Gateway → сервисы backend:** после проверки аутентификации (JWT) и прав доступа (RBAC) запрос маршрутизируется на соответствующий сервис (аутентификация, RAG, агенты, генерация отчётов и т.д.).
- **Модуль RAG ↔ векторная база данных:** Retriever выполняет запросы к векторной БД (поиск по эмбеддингам); результаты индексации документов записываются в векторную БД модулем парсинга/индексации.
- **Модуль RAG ↔ БЯМ:** Generator передаёт контекст и запрос в большую языковую модель и получает сгенерированный ответ.
- **Модуль агентов ↔ модуль RAG:** агенты (сравнение документов, анализ на соответствие и др.) могут получать данные от RAG и передавать результаты в Generator для включения в ответ.
- **Модуль парсинга и разметки → векторная БД:** обработанные документы (разбитые на блоки) и их эмбеддинги поступают в векторную БД для последующего поиска.
- **Модуль генерации отчётов:** получает данные от модуля RAG и/или агентов и формирует отчётный документ (DOCX) по шаблону.

Детализация потоков данных (форматы, протоколы) при необходимости уточняется в разделе 3 по каждому модулю и в документе «Описание информационного обеспечения».

## 3. Описание модулей (логика и функции)

### 3.1. Модуль пользовательского веб-интерфейса

**Назначение.** Модуль обеспечивает ввод запросов пользователя, загрузку документов, настройку параметров анализа и сравнения, отображение результатов и выгрузку отчётов. Пользовательский веб-интерфейс является основной точкой входа для работы с Системой (ТЗ п. 4.1.1). Описание даётся на уровне экранов и действий пользователя; логика не привязана к конкретному фреймворку (React, Vue.js или аналог).

**Состав и архитектура (SPA + BFF).** Модуль реализуется в виде одностраничного приложения (SPA) и сервера BFF (Backend for Frontend). BFF — минимальный сервер приложения, который: (1) раздаёт статические файлы SPA (HTML, JS, CSS); (2) отдаёт конфигурацию OIDC по запросу GET `/config.json` (discovery URL, client_id, redirect_uri и при необходимости базовый URL API), формируя её из переменных окружения, чтобы один и тот же билд SPA мог работать в разных окружениях (dev/prod) без изменений кода; (3) обрабатывает обмен кода авторизации на токен (POST `/auth/exchange`): принимает от SPA код, code_verifier и redirect_uri и выполняет обмен с IdP (token endpoint); (4) проксирует запросы к API: запросы SPA к тому же origin по пути `/api/*` BFF пересылает в API Gateway (Kong), подставляя JWT в заголовок `Authorization: Bearer <token>`. Браузер не обращается напрямую к IdP (кроме редиректа на страницу входа) и к Kong — только к BFF. Реализация BFF и сценарии входа описаны в документации IAM: iam/docs/frontend.md, iam/docs/auth-flow.md.

**Вход пользователя (OIDC, Authorization Code + PKCE).** Вход выполняется через единый вход (OIDC). (1) Пользователь открывает SPA по адресу BFF (например `http://localhost:3000/` или адрес развёртывания). (2) При необходимости входа SPA перенаправляет браузер на IdP (Authentik) с параметрами протокола Authorization Code + PKCE (code_challenge, state, redirect_uri, client_id, scope). (3) После успешной аутентификации IdP редиректит браузер на callback BFF (например `/callback?code=...&state=...`). (4) SPA передаёт код и code_verifier на BFF (POST `/auth/exchange`); BFF обменивает их на access_token (и при необходимости refresh_token) через token endpoint IdP. (5) Дальнейшие запросы к функциональности Системы (поиск, загрузка документов, отчёты и т.д.) SPA выполняет к тому же origin (BFF); BFF проксирует их в Kong с Bearer-токеном. Kong проверяет JWT (п. 3.2) и передаёт запрос на backend. Конфигурация (URL IdP, redirect_uri, client_id) задаётся на стороне BFF переменными окружения (OIDC_DISCOVERY_URL или AUTHENTIK_BASE_URL + OIDC_APP_SLUG, OIDC_REDIRECT_URI, OIDC_CLIENT_ID, KONG_INTERNAL_URL для проксирования) и не хранится в коде SPA. Значение redirect_uri должно совпадать с одним из Redirect URIs, настроенных у провайдера в Authentik; при несовпадении IdP возвращает ошибку invalid_grant (см. iam/docs/authentik.md).

**Основные сценарии (после аутентификации).** Инициация семантического поиска; загрузка документов (DOCX, PDF, изображения) для анализа или сравнения; настройка параметров (тип документа, объект сравнения, область поиска); просмотр результатов анализа, отчётов о различиях и рекомендаций; выбор шаблона отчёта и выгрузка результата в формате DOCX; управление составом документов в хранилище (для пользователей с соответствующими правами). Визуальный редактор разметки позволяет при необходимости корректировать границы логических блоков документа, сформированные парсером. Все эти действия выполняются через вызовы API (п. 3.2), проходящие через BFF с передачей JWT.

**Взаимодействие с backend и другими модулями.** Обмен данными с backend выполняется исключительно через вызовы API (REST over HTTPS). Цепочка запроса: браузер → BFF (тот же origin) → API Gateway (Kong) → backend-сервисы. BFF при проксировании подставляет в запрос JWT, полученный при обмене кода на токен; Kong проверяет JWT и передаёт контекст авторизации в backend (п. 3.2, 3.3). Модуль веб-интерфейса не вызывает backend и IdP напрямую из браузера (кроме редиректа на страницу входа IdP). Развёртывание: в продакшене переменные окружения задаются на сервере BFF (или через секрет-менеджер); в Authentik у провайдера должен быть зарегистрирован redirect URI приложения (например `https://your-domain.com/callback`). Сравнение архитектур статического SPA и SPA + BFF с точки зрения безопасности — iam/docs/spa-vs-bff.md.

**Входы (логические):** действия пользователя (запросы, загружаемые файлы, параметры); при первом обращении — необходимость входа (редирект на IdP, затем код и code_verifier для BFF). **Выходы:** отображение результатов поиска, отчётов и сформированных документов; выгрузка файлов (DOCX); сообщения об ошибках и состояния загрузки.

### 3.2. Модуль API Gateway

**Назначение.** Модуль API Gateway является единственной точкой входа для всех клиентских запросов к серверной части системы (веб-интерфейс через BFF, внешние клиенты, мобильные приложения). Все запросы к backend-сервисам проходят через шлюз; прямой доступ к backend в обход шлюза не предусмотрен. Шлюз обеспечивает принудительную аутентификацию (проверка JWT), проверку прав доступа (RBAC), ограничение частоты запросов (не более 100 запросов в минуту на пользователя — ТЗ п. 4.1.1), маршрутизацию трафика на соответствующие сервисы и аудит всех запросов. Выбор обусловлен требованиями ТЗ п. 4.1.1 (единая точка входа, аутентификация, RBAC, ограничение частоты, логирование) и п. 4.1.4 (контроль доступа, журналирование).

**Реализация.** В эталонной конфигурации используется Kong OSS (документ «Описание архитектуры», раздел 6.2; ТЗ п. 4.1.1). Конфигурация шлюза задаётся декларативно: файл `iam/kong/kong.yml` описывает маршруты, плагины и параметры; Kong работает в режиме DB-less (состояние не персистируется между перезапусками). Внешний доступ к API — по порту 8001 (HTTP); при необходимости добавляется HTTPS (порт 8444) и reverse proxy с TLS. Маршрут `/api` проксируется на backend-сервисы; адрес backend задаётся в конфигурации (по умолчанию в эталоне — placeholder или `http://backend:80`; для реального API — например `http://host.docker.internal:8080`). Запросы к путям, не входящим в конфигурацию (например `/` или `/other`), могут возвращать 404 — Kong обрабатывает только настроенные пути. Из контейнеров в сети Docker обращение к Kong — по внутреннему адресу (например `http://kong:8000`). Подробнее — iam/docs/kong.md.

**Проверка JWT (аутентификация).** Шлюз извлекает токен доступа из заголовка запроса (`Authorization: Bearer <token>`) и проверяет его до передачи на backend. Проверка выполняется по ключам IdP (OpenID Connect): по набору ключей JWKS или по публичному ключу в формате PEM и при необходимости по полю issuer в токене, чтобы гарантировать, что токен выдан доверенным IdP (Authentik). В Kong OSS плагин **openid-connect** недоступен (входит в Kong Enterprise), поэтому в эталонной реализации используется встроенный плагин **jwt**: в конфигурацию подставляются публичный ключ (PEM) и идентификатор ключа (kid) из IdP Authentik. JWKS провайдера доступен по URL вида `http://<authentik>/application/o/<slug>/jwks/`; скрипт `iam/kong/scripts/setup-kong-jwt-auth.sh` позволяет автоматически получить ключи из Authentik и обновить `kong.yml`. Шлюз не обращается к IdP при каждом запросе — проверка выполняется локально по ключам; синхронизация ключей выполняется при развёртывании или по процедуре обновления. При отсутствии токена или невалидной подписи/истечении срока шлюз возвращает **401 Unauthorized** и не передаёт запрос на backend; событие логируется.

#### 3.2.1. Последовательность обработки запроса

При поступлении HTTP-запроса выполняется фиксированная цепочка проверок в заданном порядке; при первой неудаче дальнейшие шаги не выполняются, клиенту возвращается ответ с кодом ошибки и выполняется логирование.

**(1) Приём запроса и извлечение JWT.** Запрос принимается по HTTPS (или HTTP на этапе без TLS); из заголовка `Authorization` извлекается токен доступа в формате JWT (схема Bearer). Если заголовок отсутствует или формат неверный, на этапе аутентификации будет возвращён 401.

**(2) Аутентификация.** Проверяется подпись и срок действия JWT по ключам IdP (Authentik) — JWKS/PEM и при необходимости issuer; обращение к IdP при каждом запросе не требуется (ключи заданы в конфигурации Kong). При отсутствии токена, невалидной подписи, неверном issuer или истёкшем сроке действия запрос не передаётся на backend; клиенту возвращается ответ **401 Unauthorized**, событие логируется. Ошибка «No credentials found for given 'kid'» указывает на то, что в Kong не подставлен актуальный ключ из JWKS Authentik — требуется выполнить скрипт настройки или обновить конфигурацию вручную (iam/docs/kong.md).

**(3) Авторизация (RBAC).** Для пары «метод HTTP + путь запроса» проверяется, разрешена ли данная операция для ролей пользователя, извлечённых из JWT. При недостаточных правах — ответ **403 Forbidden**, логирование, запрос на backend не передаётся.

**(4) Ограничение частоты запросов (rate limiting).** Проверяется, не превышен ли лимит числа запросов в минуту для данного пользователя (идентификатор из JWT). При превышении — ответ **429 Too Many Requests** (или аналог), логирование.

**(5) Маршрутизация.** При успешном прохождении всех проверок запрос вместе с контекстом авторизации (идентификатор пользователя, роли, разрешённые операции) направляется на соответствующий backend-сервис согласно правилам маршрутизации (маппинг путь + метод → целевой сервис). Ответ от сервиса возвращается клиенту через шлюз без изменения тела ответа (при успехе). Логирование выполняется для каждого запроса независимо от результата (см. п. 3.2.4). При таймауте или ошибке backend-сервиса клиенту возвращается HTTP-ответ с кодом 502 или 504, событие фиксируется в журнале.

#### 3.2.2. Ограничение частоты запросов и защита (WAF)

Ограничение частоты запросов задаётся в объёме не более 100 запросов в минуту на одного аутентифицированного пользователя (ТЗ п. 4.1.1; документ «Описание архитектуры», разделы 6.2 и 9.1). Назначение — защита backend-сервисов от перегрузки и злоупотреблений. Защита от DDoS и фильтрация вредоносного трафика обеспечивается интеграцией с WAF (Web Application Firewall) организации: трафик к API Gateway проходит через WAF или обрабатывается совместно с ним в соответствии с требованиями ТЗ («защита от DDoS с помощью WAF»). Детали развёртывания и настройки WAF в инфраструктуре Заказчика в настоящем описании не приводятся.

#### 3.2.3. Маршрутизация и передача контекста

По пути (URL) и методу HTTP запрос направляется на соответствующий backend-сервис: модуль RAG (поиск, генерация ответов), модуль агентов ИИ, модуль генерации отчётов (DOCX), сервисы загрузки и парсинга документов и т.д. Правила маршрутизации задаются конфигурацией шлюза в `iam/kong/kong.yml` (маппинг путь + метод → целевой сервис); в эталонной конфигурации единый маршрут `/api` проксируется на один backend (при необходимости правила расширяются для разных путей и сервисов). Запросы к путям, не входящим в конфигурацию, возвращают 404. Вместе с телом и заголовками запроса передаётся контекст авторизации (идентификатор пользователя, роли, разрешённые операции), сформированный на основе JWT и правил RBAC (см. п. 3.3). Backend-сервисы используют этот контекст для проверки прав при доступе к данным (в т.ч. Retriever — для фильтрации документов по доступу). Ответ от backend возвращается клиенту через шлюз без изменения тела ответа (при успехе); при таймауте или ошибке сервиса шлюз формирует ответ с кодом 5xx и фиксирует событие в журнале.

#### 3.2.4. Логирование и взаимодействие с другими компонентами

**Логирование.** Все запросы, проходящие через API Gateway, логируются с метаданными: IP-адрес клиента, идентификатор пользователя (из JWT, при наличии), время запроса, метод и путь, HTTP-статус ответа, задержка (время обработки) — в соответствии с документом «Описание архитектуры», раздел 8.4. События отказа в доступе (401, 403, 429) фиксируются отдельно и могут инициировать оповещения в SIEM. Журналы хранятся локально и передаются в систему SIEM Заказчика (по протоколу syslog или через API); срок хранения записей определяется на этапе технического проектирования с учётом регуляторных требований.

**Взаимодействие с другими компонентами.** С модулем аутентификации (п. 3.3): шлюз не вызывает IdP при каждом запросе — проверка JWT выполняется по ключам IdP (Authentik), загруженным в конфигурацию Kong; из токена извлекается контекст прав пользователя (идентификатор, роли) для RBAC и передачи в backend. С backend-сервисами: передача запроса и контекста авторизации, приём ответа; при недоступности сервиса или таймауте шлюз возвращает клиенту 5xx и фиксирует событие. С системой логирования и SIEM: передача записей журнала для аудита и оповещений.

**Входы:** HTTP-запрос (метод, путь, заголовки, тело), в т.ч. JWT в заголовке авторизации. **Выходы:** вызов backend-сервиса с передачей запроса и контекста авторизации и возврат ответа сервиса клиенту либо HTTP-ответ клиенту при ошибке (401, 403, 429, 5xx при таймауте/ошибке сервиса); для каждого запроса — запись в журнал с указанными выше метаданными.

### 3.3. Модуль аутентификации и авторизации

#### 3.3.0. Назначение и состав модуля

**Назначение.** Модуль аутентификации и авторизации выступает источником идентичности пользователя и сведений о его правах для всей системы. Он обеспечивает: установление личности пользователя при входе (аутентификация), выдачу токенов доступа (JWT) для последующих запросов к API, передачу контекста прав (роли, разрешённые операции) в API Gateway и backend-сервисы для принятия решений о доступе (авторизация). Требования ТЗ: п. 4.1.1 (единый вход, разграничение доступа), п. 4.1.4 (защита информации, MFA при необходимости).

**Состав и размещение.** Модуль реализуется не как единый монолитный сервис, а как совокупность взаимодействующих компонентов: (1) **IdP (Identity Provider)** — сервер, выдающий токены; в эталонной реализации — Authentik (или Keycloak по согласованию с Заказчиком); (2) **BFF (Backend for Frontend)** — сервер приложения, через который браузерный клиент (SPA) входит в систему и обращается к API; (3) **SPA** — одностраничное приложение в браузере, инициирующее вход и вызовы API через BFF. Логически модуль располагается между клиентом и API Gateway: клиент (через BFF) обращается к IdP для входа и получения токена; API Gateway проверяет JWT по ключам IdP и извлекает из токена контекст прав; backend-сервисы получают идентификатор пользователя и контекст прав из заголовков/контекста, сформированного шлюзом, и не обращаются к IdP напрямую.

**Входы и выходы модуля (сводно).** *Входы:* учётные данные пользователя (логин, пароль) при первичном входе; второй фактор (код TOTP или ключ FIDO2) при MFA; код авторизации и code_verifier при обмене на токен (через BFF); JWT в заголовке запроса при последующих вызовах API. *Выходы:* JWT (access_token, при необходимости refresh_token, id_token) после успешной аутентификации; контекст прав (идентификатор пользователя, роли, разрешённые операции) для API Gateway и backend-сервисов; HTTP-ответы клиенту при ошибках входа или обмена (400, 401, 403 и т.д.). Детализация по ходу описания реализации — в п. 3.3.2, 3.3.4.

#### 3.3.1. Реализация IdP (Authentik): настройка и интеграция

**Эталонная конфигурация.** В эталонной конфигурации используется Authentik 2026.2. Секреты IdP (пароль БД, секретный ключ приложения) подставляются через скрипт iam/authentik/scripts/run-authentik-with-vault.sh и хранилище Vault (детали — п. 3.3.6; см. также iam/docs/vault.md). Приложение OIDC создаётся вручную по iam/docs/authentik.md или через blueprint iam/authentik/blueprints/farmadoc-oidc.yaml.

**Провайдер и приложение в Authentik.** В Authentik создаётся провайдер типа OpenID Connect Provider и приложение (Application). У провайдера задаются: тип клиента — публичный (для браузерного SPA без client_secret); **Redirect URIs** — должны точно совпадать с теми, с которыми BFF обращается при обмене кода на токен (например `http://localhost:3000/callback`, `http://<IP>:3000/callback` при доступе по IP); подписывающий ключ; сроки действия кода авторизации, access- и refresh-токенов; scopes (openid, email, profile). Для авторизации (RBAC) в IdP настраивается включение ролей или групп пользователя в JWT (claims): Kong и backend используют эти утверждения для проверки прав; маппинг «роль/группа в IdP — утверждение в токене» задаётся в настройках провайдера или приложения. У приложения указываются имя и slug; slug используется в URL discovery (`.../application/o/<slug>/.well-known/openid-configuration`) и в JWKS (`.../application/o/<slug>/jwks/`) для Kong. **Client ID** провайдера задаётся в переменных окружения BFF (берётся в UI Authentik: Directory → Providers → выбранный провайдер).

**Интеграция с Kong и BFF.** JWKS провайдера используется API Gateway (Kong) для проверки подписи JWT; подстановка ключей в конфигурацию Kong выполняется скриптом iam/kong/scripts/setup-kong-jwt-auth.sh. Переменные окружения BFF и сценарии развёртывания описаны в п. 3.3.2 и в iam/docs/frontend.md. Подробнее по Authentik — iam/docs/authentik.md.

#### 3.3.2. Процесс аутентификации: участники, поток и конфигурация BFF

**Участники потока.** В процессе аутентификации и при последующих запросах участвуют: браузер (SPA), BFF, IdP (Authentik), API Gateway (Kong), backend. SPA не обращается напрямую к IdP (кроме редиректа на страницу входа) и к Kong — только к BFF. BFF раздаёт статику SPA, отдаёт конфигурацию OIDC (GET `/config.json`), выполняет обмен кода на токен (POST `/auth/exchange`) и проксирует запросы к API в Kong с подстановкой Bearer-токена; решений по авторизации BFF не принимает — проверка прав выполняется в Kong и backend (п. 3.3.3). Поток соответствует документации IAM: iam/docs/auth-flow.md, iam/docs/frontend.md.

**Пошаговая последовательность.** (1) Пользователь открывает SPA (доступ по адресу BFF, например `http://<host>:3000/`). (2) При необходимости входа SPA перенаправляет браузер на страницу входа IdP (Authentik) с параметрами OAuth2/OIDC: Authorization Code + PKCE (code_challenge, code_challenge_method, state, redirect_uri, client_id, scope). (3) Пользователь вводит учётные данные (логин, пароль) и при необходимости проходит MFA; после успешной проверки Authentik редиректит браузер на callback BFF с кодом авторизации и state в query-параметрах (например `/callback?code=...&state=...`). (4) SPA отправляет на BFF запрос на обмен кода на токен (POST `/auth/exchange`, тело: code, code_verifier, redirect_uri); BFF обращается к token endpoint Authentik и при успехе возвращает клиенту access_token (и при необходимости refresh_token, id_token); в токене содержатся утверждения о пользователе и ролях для последующей авторизации. (5) При последующих запросах к API клиент обращается к BFF (например GET `/api/...`); BFF подставляет JWT в заголовок `Authorization: Bearer <token>` и проксирует запрос в Kong. (6) Kong выполняет аутентификацию (проверка JWT по ключам IdP — JWKS/PEM, issuer; при отсутствии или невалидном токене возвращает 401 и не передаёт запрос на backend) и авторизацию (п. 3.3.3): для пары «метод HTTP + путь» проверяет, разрешена ли операция для ролей пользователя из JWT; при недостаточных правах возвращает 403 Forbidden и не вызывает backend. (7) При успешной проверке Kong передаёт запрос на backend с контекстом авторизации (п. 3.2); backend при необходимости выполняет дополнительную проверку прав при доступе к данным (например, Retriever возвращает только документы, доступные пользователю по ролевой модели) и при выполнении изменяющих операций.

**Конфигурация BFF и согласование с IdP.** Конфигурация клиента задаётся переменными окружения BFF: OIDC_DISCOVERY_URL (или сборка из AUTHENTIK_BASE_URL и OIDC_APP_SLUG), OIDC_REDIRECT_URI, OIDC_CLIENT_ID, при необходимости KONG_INTERNAL_URL для проксирования. **Redirect URI** должен точно совпадать с одним из Redirect URIs, настроенных у провайдера в Authentik; при несовпадении IdP при обмене кода возвращает ошибку `invalid_grant`. BFF может формировать redirect_uri по заголовку Host запроса (при доступе по IP, например `http://192.168.173.157:3000`, получится `http://192.168.173.157:3000/callback`); это значение необходимо добавить в настройки провайдера Authentik. Для корректной авторизации (RBAC) в IdP должно быть настроено включение ролей или групп в JWT (scopes или custom claims). **Client ID** берётся в UI Authentik (Directory → Providers → выбранный провайдер); без верного Client ID IdP возвращает ошибку клиента. После изменения переменных окружения BFF перезапускается (например `docker compose up -d bff`).

#### 3.3.3. Ролевая модель и проверка авторизации (RBAC)

Разграничение доступа строится на ролевой модели (RBAC). Назначение ролей пользователям выполняется средствами IdP (Authentik или иной IdP по решению Заказчика); матрица ролей и прав уточняется на этапе технического проектирования (п. 4.1.1 ТЗ). В документе «Описание архитектуры» (раздел 8.1) приведены предварительные роли:

- **Администратор** — полный доступ: настройка системы, управление пользователями, управление шаблонами отчётов и плагинами агентов.
- **Эксперт** — полный объём операций с документами: загрузка, анализ, сравнение, перевод, поиск, генерация отчётов.
- **Аналитик** — доступ на чтение и поиск, генерация отчётов; без права изменять состав документов в хранилище.
- **Пользователь с доступом только на чтение** — просмотр результатов и отчётов без права запуска анализа, сравнения и изменения данных.

На основе роли пользователя определяется, какие операции (эндпоинты API) и какие данные (в т.ч. документы в векторной БД) ему доступны. Контекст прав (идентификатор пользователя, список ролей, при необходимости — список разрешённых операций или идентификаторов документов/категорий) передаётся в backend-сервисы вместе с запросом. **Проверка авторизации** выполняется на двух уровнях: на уровне Kong — для пары «метод HTTP + путь» (доступ к эндпоинту; при недостаточных правах — 403 Forbidden); на уровне backend — при доступе к конкретным данным и при выполнении изменяющих операций (создание, обновление, удаление). Модуль RAG (Retriever, п. 3.4) использует контекст прав для фильтрации: из векторной БД возвращаются только документы, доступные пользователю по ролевой модели; попытки доступа к недоступным данным логируются. События отказа в доступе (403) фиксируются для аудита и интеграции с SIEM.

**Процесс авторизации (последовательность).** При поступлении запроса с JWT Kong в рамках цепочки плагинов (п. 3.2.1): (1) проверяет подпись и срок действия JWT; при невалидном или отсутствующем токене возвращает 401, авторизация не выполняется; (2) при валидном токене извлекает из payload JWT список ролей пользователя (имя claim задаётся при настройке и должно совпадать с настройками IdP — например `groups` или `roles`); (3) определяет метод HTTP и путь запроса; (4) сопоставляет пару (метод, путь) с таблицей правил доступа «путь и метод → разрешённые роли»; (5) при отсутствии разрешающего правила или при отсутствии у пользователя ни одной из разрешённых для данного эндпоинта ролей возвращает 403 Forbidden и не передаёт запрос на backend; (6) при успешной проверке добавляет к запросу контекст авторизации (идентификатор пользователя, роли — в заголовках или контексте) и передаёт запрос на backend. Backend использует переданный контекст для проверки прав при доступе к данным и при выполнении операций.

**Настройка авторизации.** **(1) IdP (Authentik):** в настройках провайдера OIDC или приложения включается выдача ролей или групп пользователя в JWT (custom claims, scopes или маппинг атрибутов); указывается имя claim, в котором передаются роли (например `groups`, `roles`), чтобы Kong и backend читали один и тот же формат. В Authentik это настраивается в разделе провайдера (Property Mappings, Token Exchange и т.п.); см. iam/docs/authentik.md. **(2) Kong:** правила доступа «метод HTTP + путь → разрешённые роли» задаются в конфигурации шлюза. В Kong OSS встроенный плагин ACL не привязан к claims JWT (работает с Consumer Groups Kong), поэтому для RBAC по ролям из JWT в эталонной реализации используется плагин **pre-function** (Lua): после плагина JWT в фазе access выполняется код, который читает роли из провалидированного JWT (контекст Kong или декодирование payload), метод и путь запроса и проверяет их по таблице правил; при отказе вызывается возврат 403. Таблица правил задаётся в конфиге плагина (или в отдельном фрагменте конфигурации) и хранится в репозитории (например в составе `iam/kong/kong.yml` или в подключаемом файле). Порядок плагинов на маршруте: сначала **jwt**, затем **pre-function** (проверка RBAC). При изменении матрицы «роль — эндпоинты» обновляется конфигурация Kong (таблица правил) и при необходимости настройки IdP (состав ролей в JWT). Подробнее по настройке Kong — iam/docs/kong.md.

#### 3.3.4. Взаимодействие с другими компонентами

**BFF и IdP (Authentik).** BFF обращается к IdP при обмене кода авторизации на токен: отправляет POST-запрос на token endpoint IdP (URL из discovery) с параметрами code, code_verifier, redirect_uri, client_id (и при необходимости client_secret для confidential-клиента). IdP проверяет код и PKCE и возвращает access_token, при необходимости refresh_token и id_token. При неверном redirect_uri или истёкшем/уже использованном коде IdP возвращает ошибку (например `invalid_grant`); BFF передаёт её клиенту (например 400 Bad Request). Браузер к token endpoint IdP напрямую не обращается — только редирект на страницу входа и обратно на callback BFF.

**BFF и API Gateway (Kong).** Запросы к API от SPA приходят на BFF (тот же origin, например `/api/...`); BFF подставляет в заголовок `Authorization: Bearer <access_token>` текущий JWT пользователя и проксирует запрос на Kong (KONG_INTERNAL_URL). Kong воспринимает запрос как от клиента с Bearer-токеном; проверка JWT и маршрутизация выполняются по п. 3.2. Ответ backend возвращается через Kong в BFF и далее в браузер. Браузер к Kong напрямую не обращается.

**API Gateway и IdP.** Kong не вызывает IdP при каждом запросе: проверка JWT выполняется локально по публичным ключам IdP (JWKS/PEM), загруженным в конфигурацию Kong при развёртывании или по скрипту (iam/kong/scripts/setup-kong-jwt-auth.sh). Из токена извлекаются идентификатор пользователя и роли для RBAC; сформированный контекст авторизации передаётся в backend вместе с запросом (п. 3.2.3). При отсутствии или невалидности токена Kong возвращает 401 и не обращается к backend.

**API Gateway и backend.** При успешной аутентификации и авторизации Kong передаёт запрос и контекст авторизации (идентификатор пользователя, роли, разрешённые операции) на соответствующий backend-сервис. Сервисы (в т.ч. Retriever в модуле RAG, п. 3.4) используют этот контекст для проверки прав при доступе к данным и при выполнении операций. При отказе в доступе (недостаточные права) backend или Kong возвращает клиенту 403; при невалидном/отсутствующем токене — 401. События фиксируются в журнале для аудита и интеграции с SIEM (п. 3.2, 3.4).

#### 3.3.5. Механизм хранения секретов и ключей

Назначение: хранение секретов (пароли БД, секретные ключи приложений, при необходимости ключи шифрования) в защищённом хранилище — HashiCorp Vault или аналог. В эталонной реализации механизм приведён на примере IdP Authentik: в Vault хранятся пароль PostgreSQL для Authentik и секретный ключ IdP; используется хранилище типа KV v2 с отдельным путём и ключами (например `postgres_password`, `secret_key`).

**Запуск и режимы.** Vault поднимается отдельным сервисом (например, в контейнере); доступ по API и при необходимости по веб-интерфейсу (порт 8200). В режиме разработки допускается одноузловая конфигурация с хранилищем в памяти и одним root-токеном; для производства предусматривается production mode с персистентным хранилищем и процедурой unseal (ключи разделения). Хранилище типа KV v2 с отдельным путём для секретов каждого сервиса.

**Получение кредов при старте.** Принят подход «подготовка кредов перед запуском стека». Перед запуском сервисов (например, `docker compose up`) выполняется скрипт на хосте или в среде CI, который по токену Vault один раз запрашивает нужные секреты и записывает их во временный файл переменных окружения (например `.env.vault`); контейнеры приложения подключают этот файл через `env_file` и не получают сам токен Vault и не обращаются к Vault по сети. Преимущества: (1) минимальные привилегии — контейнер получает только нужные ему секреты, при компрометации утечка ограничена ими; (2) токен Vault не попадает в контейнеры — его знает только хост или CI; (3) скомпрометированный контейнер не может запрашивать из Vault секреты других сервисов; (4) не требуется встраивать в каждый образ логику обращения к Vault (entrypoint), один скрипт подготовки выполняется до поднятия стека; (5) файл с кредами можно исключить из репозитория (`.gitignore`), ограничить права доступа (например `chmod 600`). Альтернативный подход — запрос кредов при старте каждого контейнера (entrypoint) — требует передачи токена Vault в контейнеры и увеличивает риски при компрометации; в типичной конфигурации Docker Compose предпочтительна подготовка кредов до запуска. Вариант с entrypoint оправдан при инфраструктуре короткоживущих токенов (например Kubernetes + Vault Agent/CSI) и жёстком требовании не писать секреты на диск хоста.

**Дополняемый список приложений и ключей в Vault.** Секреты хранятся в KV v2 по путям вида `secret/farmadoc/<сервис>`. Состав уточняется по ходу разработки технического проекта.

| Сервис / приложение | Путь в Vault (пример) | Ключи (назначение) |
|---------------------|------------------------|----------------------|
| Authentik (IdP) | `secret/farmadoc/authentik` | `postgres_password` — пароль БД IdP; `secret_key` — секретный ключ приложения (AUTHENTIK_SECRET_KEY). |
| API Gateway (Kong) | `secret/farmadoc/kong` | Ключи и пароли по мере настройки (например, для плагинов, админ-API). |
| Backend-сервисы | `secret/farmadoc/backend` | Секреты подключения к БД, API-ключи, внутренние токены — по мере введения. |
| Векторная БД / шифрование | `secret/farmadoc/vector-db` | Ключи шифрования данных при хранении (при использовании такого режима). |
| *(резерв для иных сервисов)* | `secret/farmadoc/<сервис>` | Уточняется при подключении сервиса. |

**Ротация секретов: ручная и автоматическая.** *Ручная:* (1) обновить значение секрета в Vault (через API, CLI или UI); (2) скриптом обновить файл с кредами из Vault без перезапуска сервисов (режим «только запись файла»); (3) перезапустить затронутые контейнеры. Для ротации пароля БД: сначала сменить пароль в самой СУБД (например, `ALTER USER ... PASSWORD '...'`), затем обновить значение в Vault и выполнить шаги (2)–(3); иначе приложение получит новый пароль из Vault, а в БД останется старый — подключение будет падать. Ручная ротация применяется при разовых сменах (компрометация, смена политики, вывод сотрудника). *Автоматическая:* периодичность (в т.ч. каждые 90 дней для ключей шифрования по п. 4.1.1 ТЗ) задаётся политикой при развёртывании. Варианты: по расписанию (cron / systemd timer) — задача генерирует или получает новый секрет, записывает в Vault, обновляет файл кредов и перезапускает затронутые сервисы; для паролей БД — предварительная смена в СУБД; динамические секреты Vault (движки для БД и т.п.) с TTL; интеграция с CI/CD — ротация в пайплайне развёртывания. В эталонной конфигурации репозитория автоматическое расписание может не быть предустановлено; его вводят при развёртывании.

*Примечание.* Секреты иных сервисов системы (API Gateway, модули backend, при необходимости ключи шифрования векторной БД и т.д.) хранятся по тому же принципу: отдельный путь в хранилище, подготовка кредов перед стартом или по процедуре ротации, без передачи токена Vault в рабочие контейнеры.

*Примечание.* Описание механизма будет уточняться по ходу разработки технического проекта (подключение дополнительных сервисов, политики ротации, интеграция с CI/CD).

### 3.4. Модуль RAG (Retriever и Generator)

Модуль RAG является ядром семантического поиска и генерации ответов в ИС «Фармадок» (ТЗ п. 4.1.1; документ «Описание архитектуры», раздел 6.4). Он объединяет извлечение релевантных фрагментов из локального хранилища документов (Retriever) и формирование ответа на основе этого контекста и при необходимости данных от агентов ИИ (Generator). Развёрнутое описание конвейера — подготовка данных, гибридный поиск (BM25 + векторный), слияние по RRF, переранжирование, слой рассуждений (reasoning) — приведено в документе **«Пояснительная записка к ТЗ. Система интеллектуального поиска документов»**; в настоящем пункте даётся описание логики модуля RAG и его связей с другими компонентами системы.

#### 3.4.1. Retriever (извлечение контекста)

Назначение Retriever — по запросу пользователя получить набор релевантных фрагментов документов с учётом прав доступа и передать их в Generator в качестве контекста. **(1)** Текст запроса преобразуется в векторное представление (эмбеддинг) с помощью модели эмбеддингов. **(2)** Выполняется поиск в векторной базе данных по метрике сходства (косинусное или аналог) с обязательной фильтрацией по RBAC: возвращаются только документы, доступные пользователю; контекст прав (идентификатор пользователя, роли) приходит от API Gateway (п. 3.3). События отказа в доступе к запрошенным данным логируются. **(3)** Выполняется отбор кандидатов (топ-k по релевантности). При реализации расширенного конвейера (см. документ по интеллектуальному поиску): лексический поиск (BM25), слияние результатов с векторным поиском по алгоритму RRF (Reciprocal Rank Fusion), при необходимости переранжирование с помощью Cross-Encoder для формирования финального списка кандидатов. **(4)** Отобранные фрагменты документов (контекст) с метаданными (источник, страница, раздел) передаются в Generator. **Взаимодействие:** с модулем векторной базы данных (п. 3.5) — запрос по эмбеддингу запроса и фильтрам доступа; с модулем аутентификации и авторизации — получение контекста прав через API Gateway (п. 3.3). Детальное описание алгоритма — раздел 6.1 настоящего документа.

#### 3.4.2. Generator (генерация ответа)

Назначение Generator — сформировать ответ пользователю на основе контекста от Retriever и при необходимости дополнительных данных от модуля агентов ИИ (п. 3.6). **(1)** Приём контекста (набор релевантных фрагментов документов) от Retriever и при необходимости структурированных данных от агентов (результаты сравнения документов, проверки соответствия, внешнего поиска и т.п.). **(2)** Маскирование персональных и чувствительных данных в контексте перед передачей в большую языковую модель — применяется Presidio или аналог (документ «Описание архитектуры», раздел 6.4). **(3)** Формирование промпта (запрос пользователя + контекст) и вызов большой языковой модели (БЯМ), размещённой локально на сервере организации или в доверенном облаке (ТЗ п. 4.1.1). **(4)** Обработка ответа модели: извлечение текста ответа и привязка утверждений к источникам (страницы, разделы документов). **(5)** Возврат сформированного ответа с обязательным цитированием первоисточников клиенту. Детальное описание алгоритма — раздел 6.2 настоящего документа.

#### 3.4.3. Конвейер и взаимодействие с другими модулями

Порядок выполнения: запрос пользователя поступает в модуль RAG → Retriever извлекает контекст из векторной БД с учётом RBAC → при наличии расширенного конвейера выполняется слой проверки качества контекста и/или переранжирование (см. документ «Пояснительная записка к ТЗ. Система интеллектуального поиска документов») → Generator формирует ответ на основе контекста и при необходимости данных от агентов → ответ возвращается клиенту. Модуль RAG получает от API Gateway контекст авторизации (идентификатор пользователя, роли) и передаёт его в Retriever для фильтрации документов по правам доступа; запрашивает векторную базу данных (п. 3.5) для семантического (и при гибридном поиске — лексического) поиска; при необходимости запрашивает дополнительные данные у модуля агентов (п. 3.6). Детали этапов оркестрации, гибридного поиска, RRF, Cross-Encoder, CRAG/Self-RAG приведены в документе «Пояснительная записка к ТЗ. Система интеллектуального поиска документов».

#### 3.4.4. Связь с критериями приёмки и алгоритмами

Реализация модуля RAG направлена на выполнение критериев приёмки системы (ТЗ п. 2.2): среднее время получения качественных результатов — не более 20 секунд; повышение качества анализа и оформления документов; обязательное цитирование источников в сгенерированных ответах. Ключевые алгоритмы семантического поиска и формирования ответа описаны в разделах 6.1 и 6.2 настоящего документа; полная детализация конвейера (подготовка данных, векторизация, гибридный поиск, переранжирование) — в документе «Пояснительная записка к ТЗ. Система интеллектуального поиска документов».

**Входы:** текст запроса пользователя; идентификатор пользователя или контекст прав от API Gateway (для RBAC в Retriever); при необходимости — параметры поиска (число кандидатов k, типы документов) и параметры генерации. **Выходы:** для Retriever — набор релевантных фрагментов документов с метаданными (источник, страница, раздел); для модуля в целом (Generator) — сгенерированный ответ с указанием источников (цитирование). Ключевые алгоритмы семантического поиска и формирования ответа описаны в разделе 6 настоящего документа; детализация конвейера интеллектуального поиска — в пояснительной записке по интеллектуальному поиску.

### 3.5. Модуль работы с векторной базой данных

Модуль обеспечивает хранение векторных представлений (эмбеддингов) документов и выполнение семантического поиска по сходству (k ближайших соседей, k-NN/ANN). Он является основой для работы Retriever в модуле RAG (п. 3.4). Реализация — Redis с расширениями векторного поиска или Pinecone, или аналог (ТЗ п. 4.1.1; документ «Описание архитектуры», раздел 6.5). Данные хранятся в зашифрованном виде; при индексации и при поиске соблюдаются требования к разграничению доступа (RBAC) и защите информации.

#### 3.5.1. Хранение и индексация

Назначение: принять эмбеддинги блоков документов и метаданные от конвейера индексации и сохранить их в векторной БД. Входные данные — векторные представления блоков (фрагментов) документов, сформированные моделью эмбеддингов после парсинга и при необходимости маскирования чувствительных данных (Presidio применяется до индексации — ТЗ п. 4.1.1, Architecture 6.5); метаданные включают идентификатор документа, тип документа, права доступа (для RBAC), при необходимости страницу и раздел. Обеспечивается логическое разделение по трём типам документов: **(1)** регламентирующие (правила, стандарты, фармакопейные статьи); **(2)** рабочие (инструкции по применению, ПУР, нормативные документы, изображения упаковки и т.п.); **(3)** кэш результатов внешнего поиска с ограничением времени хранения (TTL 24 часа — ТЗ, Architecture 6.5). Запись выполняется с учётом выбранного типа документа; при обновлении или удалении документа — соответствующее обновление или удаление записей в БД. Взаимодействие: конвейер индексации (модуль парсинга п. 3.7, модель эмбеддингов) передаёт данные в модуль векторной БД; маскирование персональных данных выполняется до передачи в конвейер индексации.

#### 3.5.2. Поиск по сходству

Назначение: по эмбеддингу запроса и параметрам поиска вернуть список релевантных фрагментов документов с учётом прав доступа. **(1)** Приём эмбеддинга запроса и параметров (число кандидатов k, при необходимости фильтры по типу документов) от Retriever (п. 3.4). **(2)** Контекст авторизации (идентификатор пользователя, роли), переданный вместе с запросом, используется для фильтрации — возвращаются только документы, доступные данному пользователю по RBAC. **(3)** Выполняется поиск k ближайших соседей (k-NN) или приближённый поиск (ANN) в соответствующем подмножестве БД (по типу документов и правам доступа). **(4)** Возврат списка идентификаторов документов/блоков и самих фрагментов текста (или ссылок на них) с метаданными (источник, страница, раздел) для передачи в Generator. Фильтрация по RBAC выполняется на уровне запроса к БД с передачей контекста прав от Retriever (п. 3.4.1). Взаимодействие: Retriever формирует эмбеддинг запроса и передаёт его в модуль векторной БД вместе с контекстом прав; модуль возвращает отфильтрованный список кандидатов.

#### 3.5.3. Защита данных: шифрование и управление ключами

Данные в векторной БД хранятся в зашифрованном виде (шифрование при хранении): применяется AES-256 или ГОСТ Р 34.12-2015 (ТЗ п. 4.1.1; документ «Описание архитектуры», разделы 6.5 и 8.3). Передача данных между компонентами (клиент векторной БД и сервисы) выполняется по TLS 1.3. Ключи шифрования хранятся в защищённом хранилище (HashiCorp Vault или аналог); ротация ключей — каждые 90 дней (ТЗ, Architecture). Модуль (или сервис, использующий векторную БД) при записи и чтении использует ключи из хранилища для шифрования и расшифрования. Конкретный механизм хранения и выдачи секретов описан в п. 3.3.4. Резервное копирование выполняется с шифрованием копий (ТЗ п. 4.1.3).

#### 3.5.4. Взаимодействие с другими модулями

Модуль векторной БД не инициирует запросы к другим компонентам; он предоставляет интерфейс для записи (индексация) и чтения (поиск). **Запись:** конвейер индексации (после парсинга документов — п. 3.7, построения эмбеддингов) передаёт в модуль векторной БД эмбеддинги и метаданные; маскирование персональных данных выполняется до индексации (Presidio). **Чтение:** Retriever модуля RAG (п. 3.4) передаёт эмбеддинг запроса и контекст прав; модуль векторной БД возвращает список кандидатов с учётом RBAC. Три типа документов (регламентирующие, рабочие, кэш внешнего поиска) позволяют при поиске ограничивать область запроса и применять политики хранения (в т.ч. TTL для кэша). Связь с критериями надёжности ТЗ (резервное копирование, время восстановления после сбоя) обеспечивается на уровне развёртывания; в описании ПО детали не приводятся.

**Входы:** при индексации — эмбеддинги блоков документов и метаданные (идентификатор документа, тип, права доступа, при необходимости страница/раздел); при поиске — эмбеддинг запроса, параметры (k, фильтры по типу документов), контекст прав для RBAC. **Выходы:** при индексации — подтверждение записи (успех или ошибка); при поиске — список идентификаторов и фрагментов документов (или ссылок на них) с метаданными для передачи в Retriever и далее в Generator.

### 3.6. Модуль агентов ИИ

Модуль агентов ИИ предоставляет расширяемый (плагинный) каркас для автономных агентов, выполняющих специализированные сценарии обработки документов и данных в изолированных, компонуемых единицах (ТЗ п. 4.1.1; документ «Описание архитектуры», раздел 6.6). Реализация строится на фреймворке оркестрации агентов LangChain, AutoGen или аналоге. Агенты запускаются в изолированных Docker-контейнерах с ограничением доступа к сети (белый список доменов), файловой системе и ресурсам; новые агенты подключаются через штатный интерфейс плагинов без изменения ядра системы (ТЗ п. 4.2.2). Результаты работы агентов при необходимости передаются в модуль RAG (Generator) или в модуль генерации отчётных документов (п. 3.4, 3.8).

#### 3.6.1. Оркестрация и поток выполнения задания

Оркестратор модуля агентов получает от вызывающего сервиса (backend) задание: тип операции и входные данные (документы, параметры проверки, запрос и т.п.). По типу операции выбирается соответствующий агент; задание передаётся агенту в изолированный контейнер. Агент выполняет последовательность вызовов: обращение к моделям (в т.ч. БЯМ при необходимости), парсерам документов, внутренним сервисам и при разрешённом сценарии — к внешним API из белого списка доменов. Результат (отчёт о сравнении, отчёт о несоответствиях, структурированная разметка, данные для RAG и т.д.) возвращается оркестратору. Оркестратор передаёт результат вызывающему сервису; при необходимости результат направляется в Generator модуля RAG (п. 3.4) в качестве дополнительного контекста или в модуль генерации отчётов (п. 3.8) для подстановки в шаблон DOCX. Контейнер агента по завершении задачи завершает работу; постоянное состояние вне векторной БД не сохраняется (Architecture 8.3). Взаимодействие: вызов от backend по запросу пользователя; передача данных в RAG (Generator) и в модуль генерации отчётов.

#### 3.6.2. Встроенные агенты первой очереди

В первой очереди реализации (ТЗ п. 4.2.1; Architecture 6.6, 9.1) в состав модуля входят следующие агенты. **Агент проверки соответствия (Compliance Agent):** проверка структуры, содержания и оформления инструкций по медицинскому применению (ИМП, SmPC/PIL) и иных документов на соответствие регуляторным требованиям (Минздрав, ЕЭК, ICH и т.п.) и референтным данным; формирование отчёта с указанием несоответствий. Входы: документ (или идентификатор), параметры проверки. Выходы: отчёт о несоответствиях. **Агент сравнения документов (Comparison Agent):** семантическое и формальное сравнение двух и более документов (инструкции, нормативные документы) с выявлением различий. Входы: два или более документа. Выходы: отчёт о различиях (семантических и формальных). **Агент разметки (Markup Agent):** разбиение документа на логические блоки (заголовки, разделы, таблицы) для последующей обработки БЯМ и индексации в векторной БД; может интегрироваться с визуальным редактором разметки. Входы: документ (DOCX, PDF). Выходы: структурированное представление (блоки с метаданными). Анонимизация персональных данных (Presidio) выполняется как middleware перед вызовом БЯМ и в описании архитектуры может относиться к отдельному компоненту; логически она обеспечивает маскирование ПД до передачи в любую БЯМ (в т.ч. при вызовах из агентов). Агенты второй очереди (веб-поиск, перевод, OCR) подключаются по мере реализации второй очереди (п. 3.9).

#### 3.6.3. Изоляция и проверка безопасности

Агенты выполняются в изолированных Docker-контейнерах («песочница») в соответствии с документом «Описание архитектуры», раздел 8.3, и ТЗ (п. 4.1.1, 4.2.2). **Сеть:** разрешён доступ только к заданному белому списку внутренних сервисов и внешних доменов; весь прочий исходящий трафик блокируется. **Файловая система:** монтирование только для чтения или в ограниченную область; доступа к файловой системе хоста и другим контейнерам нет. **Ресурсы:** ограничения по CPU и памяти задаются через механизмы контейнеризации (cgroups). **Жизненный цикл:** контейнер поднимается под задачу и уничтожается по завершении; постоянное состояние агента вне векторной БД не предусмотрено. **Проверка плагинов:** новые агенты, подключаемые как плагины (образы Docker), перед активацией проходят проверку безопасности — сканирование на вредоносный код (ClamAV или аналог) и при необходимости проверку на соответствие политикам (ТЗ п. 4.2.2; Architecture 6.6, 8.3). Детали развёртывания контейнеров и настройки сети приводятся в документе «Описание архитектуры и технических средств».

#### 3.6.4. Плагинный интерфейс и взаимодействие с RAG

Модуль поддерживает подключение новых агентов без изменения ядра системы через стандартный интерфейс плагинов: регистрация агента (образ Docker, метаданные), активация и деактивация, при необходимости обновление и удаление (Architecture 6.6, раздел 9). Взаимодействие с модулем RAG: агенты могут получать контекст из векторной БД (через интерфейс, предоставляемый системой) и передавать дополнительные данные в Generator (п. 3.4) для формирования ответа пользователю — например, результаты сравнения документов или проверки соответствия подставляются в итоговый ответ или в отчёт. Агенты второй очереди (веб-поиск, перевод, OCR) при их реализации подключаются через тот же плагинный интерфейс и выполняются в той же изолированной среде; их входы и выходы уточняются в документации по второй очереди (п. 3.9).

**Входы:** задание от вызывающего сервиса (тип операции, входные данные): для агента сравнения — два или более документа; для агента проверки соответствия — документ и параметры проверки; для агента разметки — документ (DOCX или PDF). **Выходы:** результат выполнения агента, передаваемый оркестратором вызывающему сервису и при необходимости в модуль RAG (Generator) или в модуль генерации отчётов: отчёт о различиях (сравнение), отчёт о несоответствиях (проверка соответствия), структурированное представление документа (разметка). Для агентов второй очереди форматы входов и выходов уточняются при их реализации.

### 3.7. Модуль парсинга и разметки документов

Модуль автоматически разбивает входящие документы на логические блоки (разделы, абзацы, таблицы и т.п.) для последующей обработки БЯМ и индексации в векторной БД (ТЗ п. 4.2.1, функция «Создание разметки документов»; документ «Описание архитектуры», раздел 6.7). Он является звеном конвейера индексации: результат парсинга и разметки передаётся на этап построения эмбеддингов и записи в векторную БД (п. 3.5). Поддерживаются форматы DOCX и PDF; при необходимости используется вывод OCR (вторая очередь). Используемые программные средства: python-docx, mammoth (DOCX), PyMuPDF (fitz), pdfplumber (PDF), LangChain text splitters или собственный сегментатор (раздел 4 настоящего документа).

#### 3.7.1. Парсинг документов (DOCX и PDF)

Назначение: извлечь из файла документа текст и структуру. **DOCX:** разбор формата OOXML — извлечение заголовков, абзацев, таблиц, списков, подписей к рисункам; при необходимости используется mammoth для сохранения сложной разметки. **PDF:** извлечение текстового слоя и позиционирования элементов (PyMuPDF/fitz, pdfplumber); учёт колонок, таблиц, сносок. Результат парсинга — плоский или иерархический список элементов с указанием типа (заголовок, раздел, подраздел, абзац, таблица, подпись к рисунку, сноска) и метаданных (уровень заголовка, порядок следования). При ошибках парсинга (повреждённый файл, неподдерживаемая структура) модуль возвращает вызывающему сервису код или описание ошибки.

#### 3.7.2. Сегментация и формирование блоков для индексации

Назначение: преобразовать результат парсинга в список логических блоков (chunks), пригодных для построения эмбеддингов и записи в векторную БД. Правила сегментации: объединение содержимого по заголовкам и разделам, ограничение размера блока (например, по числу токенов), сохранение целостности таблиц и списков. Для каждого блока формируются метаданные: тип блока, иерархия (уровень заголовка, родительский раздел), координаты в исходном документе (страница, диапазон), при необходимости идентификатор документа. Используются LangChain text splitters или собственный сегментатор (раздел 4). Результат передаётся конвейеру индексации (п. 3.5) и при необходимости агенту разметки (п. 3.6) или визуальному редактору разметки.

#### 3.7.3. Визуальный редактор разметки

В веб-интерфейсе для уполномоченных пользователей предоставляется визуальный редактор разметки (документ «Описание архитектуры», раздел 6.7; ТЗ п. 4.2.1 — «возможность ручного редактирования разметки через визуальный интерфейс»). Назначение: отображение автоматически определённых границ блоков и их ручная корректировка (объединение или разбиение блоков, уточнение границ). Сохранённая после правок разметка используется конвейером индексации при построении эмбеддингов. Модуль парсинга и разметки предоставляет API для загрузки документа, получения текущей разметки и сохранения отредактированной разметки; реализация экрана редактора входит в модуль пользовательского веб-интерфейса (п. 3.1).

#### 3.7.4. Взаимодействие с другими модулями

Модуль получает на вход документ (файл DOCX или PDF) от пользователя или от конвейера загрузки; при реализации второй очереди — распознанный текст или блоки от модуля OCR. Результат работы — структурированное представление документа и список блоков с координатами и метаданными — передаётся конвейеру индексации (п. 3.5) для построения эмбеддингов и записи в векторную БД. Агент разметки (п. 3.6) может вызывать функции парсинга и сегментации или использовать общий результат модуля. Визуальный редактор разметки (п. 3.1) запрашивает и сохраняет разметку через API backend. Чувствительные данные в тексте блоков маскируются перед индексацией (Presidio); место маскирования в конвейере (до или после сегментации) уточняется в документе «Описание архитектуры».

**Входы:** документ в формате DOCX или PDF; при реализации второй очереди — распознанный текст или блоки от модуля OCR; при сохранении правок — отредактированная разметка от веб-интерфейса. **Выходы:** структурированное представление документа (элементы с типами и иерархией); список логических блоков с координатами (страница, диапазон) и метаданными (тип блока, иерархия, идентификатор документа) для конвейера индексации и визуального редактора; при ошибке парсинга — код или описание ошибки для вызывающего сервиса.

### 3.8. Модуль генерации отчётных документов

Модуль формирует структурированные отчётные документы в формате DOCX на основе предопределённых шаблонов и данных, поступающих от модуля RAG и агентов ИИ (ТЗ п. 4.2.1, функция «Генерация отчётных документов на основе типовых шаблонов»; документ «Описание архитектуры», раздел 6.10). Шаблоны предоставляются Заказчиком или разрабатываются Подрядчиком; поддерживается несколько шаблонов, выбор при формировании отчёта выполняют администратор или уполномоченные пользователи. Реализация опирается на библиотеку python-docx или аналог (раздел 4 настоящего документа).

#### 3.8.1. Шаблоны отчётов и их выбор

Назначение: обеспечить использование типовых шаблонов и выбор нужного шаблона при формировании отчёта. Шаблоны представляют собой файлы DOCX с размеченными областями подстановки (плейсхолдеры, поля, при необходимости таблицы и повторяющиеся секции). Управление шаблонами: загрузка, хранение и при необходимости версионирование; администраторы загружают и управляют файлами шаблонов через веб-интерфейс (Architecture 6.10; ТЗ — возможность выбора шаблона администратором и пользователями). При запросе на генерацию отчёта пользователь или администратор выбирает шаблон; при наличии единственного шаблона по умолчанию выбор может быть неявным. Модуль получает идентификатор выбранного шаблона и набор данных для подстановки.

#### 3.8.2. Данные для подстановки и логика формирования отчёта

Назначение: подставить в шаблон переданные данные и сформировать итоговый документ DOCX. Входные данные — результаты анализа на соответствие регуляторным требованиям (отчёт о несоответствиях от агента проверки), сравнения документов (отчёт о различиях от агента сравнения), семантического поиска (ответ RAG с цитированием источников), при необходимости дополнительные структурированные сведения от модуля RAG (Generator) и агентов (п. 3.4, 3.6). Логика: загрузка выбранного шаблона DOCX; сопоставление полей данных с плейсхолдерами и областями шаблона по правилам, заданным в шаблоне (имена полей, структура таблиц, повторяющиеся блоки); подстановка текста и структурированных фрагментов; сохранение форматирования, структуры и требований к оформлению Заказчика (Architecture 6.10). При отсутствии значения для обязательного поля выполняется подстановка пустой строки или маркера согласно правилам шаблона; при ошибке (например, несовместимая структура данных) модуль возвращает вызывающему сервису код или описание ошибки.

#### 3.8.3. Вывод отчёта и сохранение

Сформированный документ сохраняется в формате DOCX. Далее возможны возврат файла клиенту для скачивания через веб-интерфейс и/или сохранение экземпляра отчёта в хранилище системы в соответствии с правилами развёртывания (ТЗ — «сохранение результатов анализа в формате DOCX по шаблону»). Права на генерацию отчётов определяются ролевой моделью (п. 3.3): администратор и эксперты имеют полный доступ к генерации отчётов; аналитик может формировать отчёты без права изменять состав документов в хранилище (документ «Описание архитектуры», раздел 8.1). Факт генерации отчёта (пользователь, выбранный шаблон, время) логируется для целей аудита.

#### 3.8.4. Взаимодействие с другими модулями

Модуль вызывается backend по запросу пользователя после выполнения анализа, сравнения или поиска. Данные для подстановки поступают от модуля RAG (результаты семантического поиска, сгенерированный ответ с цитированием) и от модуля агентов ИИ (отчёты проверки соответствия, отчёты сравнения документов) (п. 3.4, 3.6). Шаблоны хранятся в хранилище, доступном модулю; управление шаблонами (загрузка, удаление) выполняется через веб-интерфейс с правами администратора (п. 3.1). Связь с разделом 5 настоящего документа: сформированный файл DOCX входит в сводную таблицу выходных данных системы.

**Входы:** идентификатор выбранного шаблона отчёта (или файл шаблона DOCX при прямой передаче); набор данных для подстановки — результаты проверки на соответствие, сравнения документов, семантического поиска (ответ RAG), при необходимости дополнительные структурированные данные от агентов. **Выходы:** сформированный файл отчёта в формате DOCX (передача клиенту для скачивания и/или сохранение в хранилище); при ошибке — код или описание ошибки для вызывающего сервиса.

### 3.9. Модули второй очереди (кратко)

Функции второй очереди реализации заданы ТЗ п. 4.2.2 и документом «Описание архитектуры», раздел 9.2. Ниже приведено краткое описание модулей и подсистем второй очереди; детальная логика, входы и выходы уточняются при техническом проектировании и реализации второй очереди.

#### 3.9.1. Безопасный веб-поиск

Модуль (агент) обеспечивает контролируемый поиск по разрешённым внешним источникам (белый список доменов, настраиваемый администратором: EMA, FDA, WHO, Росздравнадзор и т.п.). Запросы, содержащие конфиденциальную информацию, не передаются вовне и обрабатываются только по внутреннему корпусу документов (фильтр конфиденциальности). Получение данных — через REST API или веб-скрейпинг в зависимости от ресурса. Результаты кэшируются в векторной БД (TTL 24 часа) и доступны для семантического поиска вместе с внутренними документами; интеграция с модулем RAG (документ «Описание архитектуры», раздел 6.11; ТЗ п. 4.2.2). Поддержка доступа к патентным и международным фармацевтическим реестрам; при необходимости — визуализация перевода проблем безопасности (риски, побочные действия) в интерфейсе.

#### 3.9.2. Перевод документов

Модуль перевода обеспечивает автоматический перевод фармацевтических документов с иностранных языков (английский, немецкий, французский и др.) на русский с учётом терминологии и регуляторных требований. Используется локальная или доверенная облачная БЯМ (ТЗ п. 4.2.2; документ «Описание архитектуры», раздел 6.9). Сохраняется структура исходного документа (разделы, таблицы, заголовки); выход — документ в формате DOCX с сохранением оформления. Поддерживается сценарий двуязычного сравнения ПУР (русская и английская версии) (документ «Описание архитектуры», раздел 9.2).

#### 3.9.3. Оптическое распознавание текста (OCR)

Модуль (агент) извлекает машиночитаемый текст из отсканированных документов, изображений и PDF без текстового слоя. Входные форматы: PDF (на основе изображений), JPEG, PNG. Распознавание выполняется для фармацевтических документов (упаковка, маркировка, отсканированные регуляторные формы). Выход — обычный текст и структурированные блоки для передачи в модуль парсинга и разметки (п. 3.7); распознанный текст индексируется в векторной БД для семантического поиска (документ «Описание архитектуры», раздел 6.8; ТЗ п. 4.2.2). Технологии: Tesseract OCR, PaddleOCR или аналог; реализация — в рамках модуля агентов (п. 3.6).

#### 3.9.4. Управление плагинами агентов ИИ

Подсистема предоставляет стандартный интерфейс регистрации, проверки и управления жизненным циклом агентов-плагинов без изменения ядра системы (ТЗ п. 4.2.2; документ «Описание архитектуры», раздел 6.12). Регистрация: приём новых агентов в виде образов Docker через API. Проверка безопасности перед активацией: статический анализ, сканирование на вредоносный код (ClamAV или аналог), при необходимости проверка на соответствие политикам. Управление жизненным циклом: активация, деактивация, обновление, удаление плагинов. Каждый плагин выполняется в отдельном контейнере с заданными ограничениями по ресурсам и сети. Каталог плагинов доступен администраторам через веб-интерфейс (документ «Описание архитектуры», разделы 6.12 и 9.2). Детали API и сценариев уточняются при реализации второй очереди.

Дополнительные сценарии второй очереди (двуязычное сравнение ПУР, сравнение ИИ-резюме ПУР с оригиналом, кросс-языковой поиск по внутренним и международным источникам) могут опираться на перечисленные выше модули; их описание приводится в документации по второй очереди и в документе «Описание архитектуры», раздел 9.2.

Форматы входов и выходов по каждому модулю второй очереди фиксируются при техническом проектировании и реализации (см. раздел 9 документа «Описание архитектуры» и последующие версии настоящего документа).

В настоящем разделе перечислены основные программные средства (библиотеки, фреймворки и внешние компоненты), применяемые при реализации модулей системы, описанных в разделе 3. Указание конкретных наименований и версий необходимо для однозначной идентификации зависимостей при сборке, сопровождении и валидации ПО в соответствии с практикой ЕСПД (ГОСТ 19 серии). Детали установки, конфигурирования и развёртывания перечисленных средств относятся к документу «Описание архитектуры и технических средств»; полный перечень технологий и версий приведён в документе «Описание архитектуры» (Architecture.md), раздел 7.

### 4.1. Серверная часть

- **Язык программирования:** Python 3.11 и выше (backend-сервисы и ИИ-пайплайны).
- **Фреймворк API:** FastAPI или функциональный аналог (реализация REST API, маршрутизация, валидация запросов).
- **Фреймворки агентов ИИ:** LangChain, AutoGen или аналоги (оркестрация агентов, цепочки вызовов, интеграция с моделями).
- **Работа с документами:** python-docx (генерация и парсинг DOCX), mammoth (парсинг DOCX), PyMuPDF (fitz), pdfplumber (парсинг PDF).
- **Эмбеддинги и векторный поиск:** библиотеки для вызова моделей эмбеддингов и клиенты векторной БД (в соответствии с выбранным стеком, см. документ «Описание архитектуры», раздел 7).
- **Разбиение текста на блоки:** LangChain text splitters или собственный сегментатор.

Назначение перечисленных средств в логике программы: обеспечение REST API, оркестрация агентов, преобразование документов в структурированный вид и в эмбеддинги, запись и поиск в векторной БД, формирование отчётных файлов.

### 4.2. Клиентская часть

- **Фреймворк SPA:** React или Vue.js (по выбору при разработке). Назначение: реализация пользовательского интерфейса, отправка запросов к API, отображение результатов и управление состоянием экранов.

### 4.3. Внешние компоненты

- **Authentik (IdP)** — аутентификация и авторизация (SSO, MFA, OAuth2/OIDC, выдача JWT); или Keycloak по согласованию с Заказчиком. Документация по настройке: iam/docs/authentik.md, iam/docs/auth-flow.md.
- **API Gateway (Kong или аналог)** — единая точка входа, проверка JWT и RBAC, ограничение частоты запросов, маршрутизация (компонент может быть внешним по отношению к коду приложения).
- **Векторная база данных (Redis/Pinecone или аналог)** — хранение эмбеддингов и поиск по сходству.
- **HashiCorp Vault** — хранение секретов и при необходимости ключей шифрования; применение в системе (в т.ч. на примере Authentik) описано в п. 3.3.4.
- **Microsoft Presidio или аналог** — маскирование персональных и чувствительных данных перед передачей в БЯМ.

Роль перечисленных компонентов в логике ПО: обеспечение идентификации, безопасности, хранения векторных данных и защиты персональных данных. Вопросы установки и развёртывания этих компонентов относятся к документу «Описание архитектуры и технических средств».

Полный перечень технологий приведён в документе «Описание архитектуры» (Architecture.md), раздел 7.

|----------|------------------|------------|
| Пользователь | Текст запроса поиска, параметры анализа/сравнения | Семантический поиск, запуск анализа и сравнения документов |
| Пользователь | Документы (DOCX, PDF, изображения) | Загрузка для анализа, сравнения, индексации |
| Администратор / пользователь | Шаблоны отчётов (DOCX) | Генерация отчётных документов по выбранному шаблону |
| Администратор | Конфигурация (роли, права, белые списки источников и т.п.) | Настройка системы и политик доступа |
| Внешние системы (вторая очередь) | Данные из разрешённых источников (по API или веб-скрейпингу) | Кэширование результатов веб-поиска в векторной БД |

Детальное описание форматов (структура JSON/XML запросов и ответов, структура файлов) приводится в документе «Описание информационного обеспечения».

### 5.2. Выходные данные

| Выход | Получатель | Назначение |
|-------|------------|------------|
| Результаты семантического поиска (список фрагментов документов, сгенерированный ответ) | Пользователь (через веб-интерфейс) | Ответ на запрос поиска |
| Отчёты анализа/сравнения (структурированные данные и/или DOCX) | Пользователь | Результаты проверки документов на соответствие, отчёт о различиях |
| Сгенерированные документы (DOCX) | Пользователь | Выгрузка отчёта по шаблону |
| Сообщения об ошибках | Пользователь / система логирования | Информирование об ошибках выполнения операций |
| Журналы (логи запросов, отказов в доступе, системных событий) | Система мониторинга / SIEM | Аудит и расследование инцидентов |

Детальное описание форматов выходных данных (в т.ч. структура DOCX, форматы логов) — в документе «Описание информационного обеспечения».

## 6. Алгоритмы (ключевые)

### 6.1. Алгоритм семантического поиска

1. Получение текста запроса и идентификатора пользователя.
2. Преобразование запроса в векторное представление (эмбеддинг) с помощью модели эмбеддингов.
3. Поиск в векторной базе данных по метрике сходства (например, косинусное) с учётом прав доступа (RBAC): возвращаются только документы, доступные пользователю. При отказе в доступе к запрошенным данным — логирование события.
4. Отбор кандидатов (топ-k по релевантности). При использовании гибридного поиска (лексический + векторный) — слияние результатов (например, по алгоритму RRF) и при необходимости переранжирование (Cross-Encoder).
5. Передача отобранных фрагментов документов (контекста) в компонент Generator для формирования ответа.

Детализация конвейера (в т.ч. гибридный поиск, переранжирование) приведена в документе «Пояснительная записка к ТЗ. Система интеллектуального поиска документов».

### 6.2. Алгоритм формирования ответа (RAG)

1. Получение контекста от Retriever (набор релевантных фрагментов документов) и при необходимости дополнительных данных от агентов ИИ.
2. Проверка и при необходимости маскирование персональных и чувствительных данных в контексте (Presidio или аналог) перед передачей в БЯМ.
3. Формирование промпта для большой языковой модели (запрос пользователя + контекст) и вызов БЯМ.
4. Обработка ответа модели: извлечение текста ответа и привязка к источникам (страницы, разделы документов).
5. Возврат сформированного ответа с цитированием источников клиенту.

### 6.3. Алгоритм индексации документа

1. Получение документа (файл DOCX, PDF или результат OCR) и метаданных (тип документа, права доступа и т.п.).
2. Парсинг документа и разбиение на логические блоки (модуль парсинга и разметки).
3. При необходимости маскирование чувствительных данных в тексте блоков.
4. Преобразование каждого блока (или объединённых блоков) в векторное представление (эмбеддинг) с помощью модели эмбеддингов.
5. Запись эмбеддингов и метаданных в векторную базу данных с учётом логического разделения по типам документов (регламентирующие, рабочие, кэш внешнего поиска) и разграничения доступа.

Блок-схемы указанных алгоритмов при необходимости добавляются в приложения к настоящему документу или в сопроводительные материалы по валидации ПО.
